# 文旅 GraphRAG 项目面试问答：项目整体与多智能体架构

## 一、项目整体

### 1. 你用 1 分钟介绍一下这个项目，解决了什么问题？

这个项目是一个面向文旅场景的智能问答与旅游规划原型系统，主要解决游客在做行程规划时遇到的信息分散、多景点关联查询困难、路线和票务规则组合复杂的问题。

传统 RAG 对单点问题，比如“某个景点门票多少钱”，效果通常还可以。但文旅问题经常是组合型问题，比如“三天游玩象鼻山、两江四湖和漓江怎么安排”，它同时涉及景点介绍、票价、交通、地理邻近关系、游玩顺序等信息。单纯依赖向量相似度检索，容易漏掉实体之间的结构化关系。

所以我在普通 RAG 的基础上引入了 Neo4j 知识图谱和多智能体编排。离线侧负责解析 DOCX、PDF、Markdown 等文旅资料，完成切块、向量化、BM25 索引和知识图谱构建；在线侧由 LLM 先做意图识别和任务拆解，再路由到局部图谱检索、向量检索或全局社区摘要检索，最后汇总生成回答或规划建议。

目前这个项目更偏原型系统，核心 GraphRAG 检索链路、多智能体任务编排、缓存和评估模块已经实现，但还没有完全产品化部署。

### 2. 为什么文旅场景需要 GraphRAG，而不是普通 RAG？

普通 RAG 的核心是从文档块里找语义相似内容，它适合回答单点事实问题，但对实体之间的关系不敏感。文旅场景里，很多问题不是只查一个文本片段，而是需要把多个实体和关系串起来。

比如用户问：

“两江四湖附近有哪些码头？夜游票价是多少？附近还有什么景点可以一起安排？”

这个问题里包含景点、码头、票务、地理邻近和路线组合。相关信息可能散落在多个文档中。如果只靠向量检索，可能召回到某个票价片段，但漏掉码头和邻近景点的关系。

GraphRAG 的价值是把文档中的景点、地点、交通工具、票务价格、规则政策等抽成实体，把位于、毗邻、价格为、包含、通往、适用于等抽成关系，写入 Neo4j。查询时可以先识别核心实体，再查一跳或多跳邻居，把结构化关系和向量文档片段一起交给 LLM。

这样系统不只是“找相似文本”，还可以利用实体关系辅助回答多景点关联、票价规则、交通路线和整体旅游规划类问题。

### 3. 这个系统的整体架构是什么？离线和在线分别做什么？

整体上我把系统分成离线数据构建层和在线问答决策层。

离线数据构建层主要做四件事：

1. 文档解析：使用 Docling 或 DeepDoc 解析 PDF、DOCX、Markdown 等景区资料，尽量保留标题、表格和结构信息。
2. 文本切块：把解析后的 Markdown 按标题和语义结构切成 chunk，避免完全按固定长度切导致表格或章节被截断。
3. 检索索引构建：一方面用 embedding 模型构建 FAISS 向量库，另一方面用 jieba + BM25 构建关键词检索索引。
4. 图谱构建：通过 LLM 从文本块中抽取实体和关系，写入 Neo4j，并做一定的数据清洗，比如去除自环、孤立节点和归一化部分关系类型。

在线问答层主要包括：

1. Planner：分析用户问题，拆解成 local_search、global_search 等任务。
2. Executor：根据任务类型调用对应检索工具。
3. Reporter：把多个检索任务的证据汇总成结构化报告或回答。
6. Cache：对重复或相似问题做缓存复用，减少重复推理成本。

### 4. 用户输入一个问题后，从进入系统到最终回答，中间完整链路是什么？

以“三天内游玩象鼻山、两江四湖、漓江，该怎么安排？”为例，在线链路大致是：

1. 用户输入问题。
2. 多智能体编排器创建一个执行状态对象，保存原始问题、任务计划、执行记录和最终报告。
3. Planner 调用 LLM，根据用户问题生成任务图。例如拆出景点信息查询、路线或行程建议查询、整体规划总结等任务。
4. Worker 读取任务图，根据任务类型调用对应工具。具体事实类问题会走 local_search，宏观总结类问题会走 global_search。
5. local_search 内部会先抽取问题中的核心实体，再从 Neo4j 查询相关子图上下文，同时调用向量检索和 BM25 混合召回文档片段，并用 reranker 选出更相关的内容。
6. global_search 会读取 Neo4j 中的社区摘要，用于回答整体性、总结性问题。
7. 每个任务执行完后，会生成 execution record，记录输入、输出、耗时和错误信息。
8. Reporter 根据所有执行记录生成报告大纲，再逐章节或整体生成最终回答。
9. 如果开启缓存，最终结果会写入缓存；后续相同或高度相似的问题可以复用结果。

这个链路的核心思想是：复杂问题不直接交给一个 RAG prompt，而是先拆解，再分别检索，最后基于证据汇总。

### 5. 这个项目中你主要负责哪些模块？哪些是你自己实现的？

我主要负责的是围绕 GraphRAG 和多智能体问答链路的原型实现，重点包括以下几个部分：

1. RAG 检索链路：实现文档解析、Markdown 结构化切块、FAISS 向量检索、BM25 关键词检索、RRF 融合和 CrossEncoder 重排。
2. GraphRAG 模块：设计旅游领域实体和关系 schema，基于 LLM 抽取实体关系，写入 Neo4j，并实现局部子图检索和社区摘要式全局检索。
3. 多智能体编排：实现 Plan-Execute-Report 流程，用 Planner 拆任务，用 Worker 路由检索工具，用 Reporter 汇总结果。
4. 缓存模块：实现 memory、disk、hybrid 后端，以及上下文感知的缓存 key 策略和语义相似命中。
5. 评估模块：实现答案质量、检索质量、缓存性能等指标，用来分析系统效果。

其中 DeepDoc 相关代码更像是参考或集成现有文档解析能力，核心业务逻辑主要集中在 RAG、GraphRAG、多智能体编排和缓存这些模块。

### 6. 目前系统是原型还是可部署系统？最大短板是什么？

目前我会把它定位为一个本地可运行的原型系统，而不是完整生产级系统。

已经实现的部分包括：文档解析、向量检索、BM25、rerank、Neo4j 图谱构建、Local/Global GraphRAG、多智能体任务编排、缓存和评估。

但它还有几个明显短板：

1. 配置不够工程化：Ollama 地址、Neo4j 密码、本地模型路径等还存在硬编码，应该统一迁移到环境变量或配置文件。
2. 启动成本较高：部分模块在 import 阶段就会加载文档、构建向量库、加载 reranker 和连接 Neo4j，后续应该改成显式初始化和懒加载。
3. 服务化不足：目前主要是脚本和模块形式，还缺少完整 FastAPI 服务入口、鉴权、监控和部署配置。
4. 评估还不够系统：虽然有评估模块，但还需要更完整的标注数据集和消融实验，比如比较纯向量、向量+BM25、GraphRAG 的效果差异。
5. 路线规划还不是严格优化算法：目前更多是基于检索证据和 LLM 生成行程建议，还没有完整实现时间窗、距离、开放时间约束下的路径优化。

如果要继续产品化，我会优先做配置治理、服务化 API、索引持久化和评估闭环。

## 二、多智能体架构

### 1. 你说用了多智能体，具体有哪些 Agent 或角色？

这个项目里的多智能体不是让多个 LLM 自由对话，而是采用相对可控的角色分工，核心是 Plan-Execute-Report。

主要角色有三个：

1. Planner：负责任务规划。它读取用户问题，把复杂问题拆成多个可执行子任务，并生成任务图。
2. Worker 或 Executor：负责任务执行。它根据任务类型调用对应工具，比如 local_search 或 global_search。
3. Reporter：负责结果汇总。它读取各个任务的执行记录，生成报告大纲和最终回答。

此外，在 `graphrag_agent` 里还有基于 LangGraph 的 HybridAgent，它把 local_hybrid_search 和 global_macro_search 封装成工具，让 LLM 根据问题决定是否调用工具。这个是另一条更偏工具调用和流式输出的 Agent 实现路线。

### 2. Planner、Worker、Reporter 分别做什么？

Planner 的职责是把用户问题拆成结构化任务。它会调用 LLM，根据 prompt 生成 TaskGraph。每个任务包含 task_id、task_type、description、priority、depends_on、status 等字段。任务类型主要包括 local_search、global_search 和预留的 reflection。

Worker 的职责是调度和执行任务。它会检查任务依赖，找到当前可以执行的任务，然后根据 task_type 从工具注册表里找到对应工具。例如 local_search 会调用局部图谱和混合检索，global_search 会调用社区摘要检索。

Reporter 的职责是把多个任务的结果组织成最终回答。它先根据用户问题和 execution records 生成报告大纲，再基于证据生成章节内容或结构化回答。

这样做的好处是每个角色的输出边界比较清楚：Planner 只规划，Worker 只检索，Reporter 只汇总，降低了复杂问题直接由一个 prompt 处理时的不稳定性。

### 3. 任务拆解的结果是什么格式？怎么保证 LLM 输出可解析？

任务拆解结果是一个 JSON 格式的 TaskGraph，大致结构是：

```json
{
  "nodes": [
    {
      "task_id": "task_001",
      "task_type": "local_search",
      "description": "查询象鼻山的景点信息和游玩建议",
      "priority": 1,
      "depends_on": [],
      "status": "pending"
    }
  ],
  "execution_mode": "sequential"
}
```

代码里用 Pydantic 定义了 `TaskNode`、`TaskGraph` 和 `PlanSpec`，用来约束字段结构。Planner 在解析 LLM 输出时，会先尝试从 Markdown 代码块中提取 JSON，如果没有代码块，就查找第一个 `{` 和最后一个 `}` 之间的内容，再用 `json.loads` 解析。

如果解析失败，系统不会直接崩溃，而是生成一个默认的 local_search 任务作为兜底。这样至少可以保证用户问题进入基础检索链路。

不过这个方案还不是最强约束。更好的方式是后续接入模型的结构化输出、function calling 或 LangChain 的 Pydantic parser，进一步降低 JSON 格式错误。

### 4. 如果 Planner 拆错任务，系统怎么处理？

目前系统有基础兜底，但还没有完全成熟的自我修正机制。

已有处理包括：

1. JSON 解析失败时，会生成默认 local_search 任务，避免流程中断。
2. Worker 执行任务时，如果工具调用异常，会把错误写入 execution record，而不是直接让整个程序崩溃。
3. Reporter 会基于已有 execution records 生成回答，如果证据不足，prompt 会要求如实说明。

但如果 Planner 逻辑上拆错了，比如把一个具体票价问题误判成 global_search，当前系统还不能稳定自动纠正。项目里虽然预留了 reflection 任务类型，但还没有完整实现反思和重规划工具。

如果继续优化，我会增加两个机制：

1. 路由校验：在执行前对任务类型做一次轻量规则或 LLM 二次判断。
2. 结果反思：如果检索结果为空或置信度低，自动回退到另一类检索工具，比如 local_search 和 global_search 互相兜底。

### 5. 任务之间有依赖关系吗？怎么调度？

任务结构里有 `depends_on` 字段，用来表示依赖关系。调度时，Worker 会调用 `get_ready_tasks`，只取状态为 pending 且依赖任务都已完成的任务。

当前实现主要是顺序执行。流程大致是：

1. 读取所有任务。
2. 找出依赖已满足的 ready tasks。
3. 按 priority 排序。
4. 逐个执行任务。
5. 执行成功后把任务标记为 completed。
6. 循环直到所有任务完成，或发现没有可执行任务但仍有任务未完成，这时认为可能存在死锁或依赖错误。

这个设计为后续并行执行留了接口，因为 TaskGraph 里有 execution_mode 字段，但当前实现重点还是稳定地跑通顺序链路。

### 6. 为什么不用一个大 Prompt 直接回答，而要做任务拆解？

主要原因是文旅问题经常是复合型任务。一个大 Prompt 直接回答会有几个问题：

1. 检索目标不清晰：复杂问题里可能同时包含景点、路线、票价、时间安排，单次检索容易只命中其中一部分。
2. 上下文混杂：把所有召回内容直接塞给 LLM，容易导致信息冗余、冲突和遗漏。
3. 难以路由不同检索方式：具体事实适合 local_search，宏观总结适合 global_search，单一路径难以同时兼顾。
4. 难以追踪错误：拆成任务后，可以看到每个子任务的输入、输出、耗时和错误，更容易调试。

任务拆解的意义是把复杂问题变成多个可控子问题。例如“三天游桂林”可以拆成景点信息查询、交通关联查询、行程顺序建议和最终汇总。每个子任务分别检索，再由 Reporter 汇总，整体可靠性比单次 prompt 更高。

### 7. `local_search` 和 `global_search` 是怎么路由的？

项目里有两种路由方式。

第一种是在 `agent_system` 里，由 Planner 直接生成任务类型。对于具体事实类问题，Planner 会生成 local_search；对于宏观总结、整体分析类问题，会生成 global_search。

第二种是在测试脚本和 HybridAgent 里，用 LLM 或工具描述来判断。比如：

1. local_search 适合查询具体实体、票价、路线、交通、开放时间等微观事实。
2. global_search 适合查询整体趋势、综合介绍、跨景区总结、社区级摘要等宏观问题。

local_search 的底层实现会组合两类上下文：Neo4j 局部子图上下文和向量文档片段。global_search 则主要读取 Neo4j 中 Community 节点的 summary，用于生成全局视角回答。

实际面试中我会强调：当前路由主要依赖 LLM 判断和任务类型，属于原型实现。后续可以增加规则特征，比如是否包含“总结、整体、比较、规划”等宏观意图词，或者是否包含明确景点、票价、路线等实体类意图词，来提升稳定性。

### 8. `reflection` 有没有真正实现？如果没有，为什么简历里不强调？

`reflection` 在任务类型里是预留的，但目前没有完整接入工具注册和执行逻辑。因此我不会在简历里重点写“反思智能体”或“自我纠错智能体”。

目前真正跑通的是 Planner、Worker、Reporter，以及 local_search 和 global_search 两类工具。reflection 更适合作为后续增强方向，比如用于检查检索证据是否足够、回答是否忠于上下文、是否需要补充检索。

如果面试官问到，我会如实说明：项目里预留了 reflection 的任务类型，但当前核心实现集中在任务拆解、检索路由和报告汇总，反思闭环还没有完整产品化。我认为这种说法比夸大已经实现更稳妥。

## 三、RAG 检索链路

### 1. 你的普通 RAG 流程是什么？

当前版本可以这样答：

我的普通 RAG 流程主要包括离线建库和在线查询两部分。

离线阶段，系统先读取文旅资料，比如 Markdown、DOCX、PDF 等文档。对于 DOCX、PDF 这类复杂格式，会通过 Docling 转成 Markdown，尽量保留标题、段落和表格结构。然后对 Markdown 进行结构化切块，得到一组 `Document` 对象。之后使用本地 embedding 模型把 chunk 向量化，构建 FAISS 向量库，同时用 jieba 对 chunk 分词，构建 BM25 索引。

在线阶段，用户问题会先经过 query rewrite，把口语化问题改写成更适合检索的表达。然后系统分别做向量召回和 BM25 召回，再用 RRF 对两路结果融合。融合后的候选文档会进入 CrossEncoder rerank，选出最相关的 top-k 片段。最后把这些片段拼成上下文，交给本地 LLM 生成回答。

优化后可以这样答：

如果继续优化，我会把当前脚本式 RAG 流程拆成更清晰的 Pipeline：`Loader -> Parser -> Chunker -> Indexer -> Retriever -> Reranker -> Generator`。同时把 FAISS 索引持久化，避免每次启动都重新解析文档和构建索引。在线侧会增加检索置信度判断、引用片段返回和答案忠实性校验，让系统更接近可部署服务。

如何优化：

1. 把文档解析、切块、索引构建从 import 阶段移到显式初始化命令。
2. 使用 FAISS `save_local/load_local` 持久化索引。
3. 每个 chunk 增加 metadata，比如 source、page、section、chunk_id。
4. 最终回答中返回引用来源，便于验证答案。
5. 增加检索分数阈值和无答案兜底逻辑。

### 2. 文档是怎么切块的？为什么按 Markdown 标题切，而不是固定长度？

当前版本可以这样答：

项目里有两种切块方式。早期版本对 Markdown 文本使用 `RecursiveCharacterTextSplitter`，按固定 chunk size 和 overlap 切分。后面针对 DOCX、PDF 等结构化文档，我改成先用 Docling 转 Markdown，再用 `MarkdownHeaderTextSplitter` 按标题层级切分。

这样做的原因是文旅资料里有很多表格、票价规则、景点介绍和分级标题。如果完全按字符数硬切，可能把一个票价表、交通规则或章节说明切断，导致检索出来的上下文不完整。按 Markdown 标题切块可以保留语义结构，让一个 chunk 更接近一个完整主题，比如某个景区的票务、交通或开放时间。

不过当前实现还比较简单，如果某个章节特别长，代码里只是注释提到可以二次切分，还没有做很完善的长度控制。

优化后可以这样答：

优化后我会采用“标题语义切分 + 长块二次递归切分”的混合策略。先按 Markdown 标题保持章节语义，如果某个章节超过模型上下文或检索粒度阈值，再按段落、表格、列表进行二级切分。这样既保留结构，又避免 chunk 过长影响召回和 rerank。

如何优化：

1. 给 `chunk_markdown` 增加最大长度参数。
2. 对超长标题块使用 `RecursiveCharacterTextSplitter` 二次切分。
3. 对表格单独保护，避免把同一张表切到多个 chunk。
4. 在 metadata 中记录标题路径，比如 `景区 -> 票务 -> 儿童票政策`。
5. 用 chunk_id 建立文档块和图谱实体之间的关联。

### 3. chunk size 和 overlap 怎么选？

当前版本可以这样答：

当前项目里固定长度切分的默认配置大概是 `chunk_size=500`、`chunk_overlap=50`，这是一个经验值。选择 500 左右主要是为了让每个片段包含足够完整的信息，同时又不会过长，避免向量检索时一个 chunk 里混入太多主题。overlap 设置 50 是为了减少边界信息丢失，比如一句交通规则刚好跨越两个 chunk。

对于 Docling 转 Markdown 后的结构化切分，主要按标题切，所以 chunk 长度不是严格固定的。

优化后可以这样答：

更好的做法是根据文档类型和评估结果调参，而不是固定经验值。比如票价表、问答类文档、景区介绍类文档适合不同粒度。最终可以通过检索 recall、rerank 后命中率、答案正确率来选择 chunk size。

如何优化：

1. 对不同文档类型配置不同切块策略。
2. 统计 chunk 长度分布，过滤过短和过长 chunk。
3. 做消融实验：300、500、800、1000 等 chunk size 对比。
4. 用标注问题集评估不同参数下的召回率和答案质量。

### 4. 向量检索用了什么 embedding 模型？为什么选它？

当前版本可以这样答：

项目里主要使用的是 `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2` 的本地模型路径。选择它的原因是项目资料和用户问题主要是中文，同时也可能包含少量英文或数字信息，这个模型支持多语言，体量相对较小，适合本地 CPU 环境快速实验。

这个选择更偏工程可用性和本地部署便利性，不是说它一定是效果最优的中文 embedding 模型。

优化后可以这样答：

如果追求更好的中文检索效果，我会考虑替换成更强的中文或多语言 embedding 模型，比如 BGE 系列或专门面向中文语义检索的模型。同时用项目自己的问答集评估 embedding 模型，而不是只凭模型知名度选择。

如何优化：

1. 对比 `paraphrase-multilingual-MiniLM`、`bge-small-zh`、`bge-base-zh`、`bge-m3` 等模型。
2. 用同一批文旅问题测试 top-k recall。
3. 评估模型大小、推理速度、召回效果之间的平衡。
4. 如果部署资源允许，用更强 embedding；如果资源紧张，保留轻量模型。

### 5. 为什么还要 BM25？向量检索不是够了吗？

当前版本可以这样答：

向量检索擅长语义相似，但对精确词、数字、票价、专有名词不一定稳定。文旅场景里有很多实体名、票价、公交线路、码头名称、政策条件，比如“100路公交车”“210元”“1.2米以下儿童”等。这些内容 BM25 往往更敏感。

所以项目里保留了 BM25 关键词召回，再和向量召回融合。向量检索负责语义泛化，BM25 负责精确匹配，两者互补。

优化后可以这样答：

后续我会把 BM25 做得更适合中文文旅场景，比如加入自定义词典，把“两江四湖”“象鼻山”“日月双塔”“夜游船票”这类词固定成旅游领域词，减少 jieba 错切。同时对数字、价格、公交线路做规则增强。

如何优化：

1. 加旅游领域自定义词典。
2. 对价格、时间、公交线路、身高规则做正则抽取并提升权重。
3. 对 BM25 和向量召回设置不同 top-k，再通过评估选择最佳比例。
4. 对命中实体名的 chunk 做轻量 boost。

### 6. RRF 融合怎么做？公式是什么？

当前版本可以这样答：

项目里使用 RRF，也就是 Reciprocal Rank Fusion，来融合向量召回和 BM25 召回结果。它不直接比较两种检索器的原始分数，而是根据排名来融合，因此比较适合不同检索器分数尺度不一致的场景。

公式可以表示为：

```text
score(d) = sum(1 / (k + rank_i(d)))
```

其中 `rank_i(d)` 是文档 d 在第 i 个检索器结果中的排名，`k` 是平滑常数，项目里常用 60。一个文档如果同时在向量检索和 BM25 中排名靠前，它的融合分数就会更高。

当前实现是分别取向量 top-k 和 BM25 top-k，然后按文档内容去重，用 RRF 分数排序，得到融合候选集。

优化后可以这样答：

后续可以把 RRF 从简单函数升级成可配置检索融合器，支持不同召回器权重。例如在票价、公交线路这类精确查询中提高 BM25 权重，在泛化描述类问题中提高向量权重。

如何优化：

1. 支持 weighted RRF。
2. 根据 query 类型动态调整向量和 BM25 权重。
3. 使用 chunk_id 去重，而不是仅按 page_content 去重。
4. 记录每个候选 chunk 的来源召回器和排名，便于调试。

### 7. CrossEncoder rerank 在哪里用？解决了什么问题？

当前版本可以这样答：

CrossEncoder rerank 用在混合召回之后。向量检索和 BM25 会先召回一批候选文档，RRF 融合后仍然可能有一些不够相关的片段。CrossEncoder 会把 query 和每个候选 chunk 拼成 pair，直接判断二者相关性，然后重新排序，取 top-k 作为最终上下文。

项目里使用的是 `BAAI/bge-reranker-base` 的本地模型路径。它的作用是提升最终进入 LLM 的上下文质量，减少噪声片段。

优化后可以这样答：

后续我会让 rerank 更可控，比如只对候选 top-20 做 rerank，避免成本过高；同时记录 rerank 分数，用于判断是否有足够证据回答。如果 top 分数过低，就触发“不知道”或补充检索。

如何优化：

1. 限制 rerank 候选数量，避免延迟过高。
2. 保存 rerank 分数，作为答案置信度参考。
3. 对不同问题类型设置不同 top-k。
4. 如果 rerank 分数低，触发 fallback 检索或拒答。

### 8. rerank 会不会带来性能问题？怎么权衡？

当前版本可以这样答：

会有性能开销。CrossEncoder 和 embedding 不同，它需要对每个 query-document pair 做深度交互计算，所以比向量相似度检索慢。当前项目为了保证回答质量，在召回阶段先控制候选数量，比如向量和 BM25 各取 top-10，再 rerank 取 top-3。

这个权衡是：先用便宜的召回方法扩大候选范围，再用较贵的 rerank 精排少量候选。对于原型系统，这个延迟可以接受。

优化后可以这样答：

如果要上线，我会把 rerank 做成可选策略。简单事实问题可以减少 rerank 数量，复杂规划问题才使用更强 rerank。同时加入缓存，缓存 query 的检索结果和 rerank 结果，降低重复请求成本。

如何优化：

1. 根据 query 难度决定是否 rerank。
2. 对候选数量设置上限。
3. 使用批量 predict，减少模型调用开销。
4. 缓存 rerank 结果。
5. GPU 可用时把 reranker 放到 GPU。

### 9. 如果检索不到答案，系统怎么避免幻觉？

当前版本可以这样答：

项目里有两个层面的控制。

第一是检索上下文校验。部分 RAG 脚本里会判断最终文档是否为空、总长度是否过短，如果上下文不足，就返回“当前知识库中没有足够信息回答该问题”。

第二是 prompt 约束。生成回答时会明确要求 LLM 严格基于上下文回答，如果上下文中没有答案，就说明不知道或资料未提及。

不过当前控制还比较基础，主要依赖上下文长度和 prompt 约束，还没有完整的事实一致性检测。

优化后可以这样答：

后续我会结合 rerank 分数、召回来源数量和答案引用做更严格的无答案判断。如果最终答案中的关键事实无法映射到检索片段，就要求模型重答或拒答。

如何优化：

1. 用 rerank top score 作为证据置信度。
2. 要求答案输出引用 chunk_id。
3. 生成后做 citation check，检查答案关键句是否有来源。
4. 对无来源的结论进行删除或标记为“资料未提及”。
5. 对实时性问题，比如天气、实时票价，明确提示当前知识库不覆盖。

### 10. 你怎么评估检索结果好坏？

当前版本可以这样答：

项目里有 evaluation 模块，主要设计了检索 precision、retrieval utilization、retrieval latency、chunk utilization 和缓存性能等指标。

检索 precision 主要看回答中引用或使用的实体，有多少确实来自检索结果，用来衡量检索证据是否支撑答案。retrieval utilization 看检索出来的实体或片段有多少被最终回答真正使用。latency 用来衡量检索耗时。chunk utilization 则关注最终回答是否利用了检索到的 chunk 内容。

目前这些指标更偏原型评估，还需要更多人工标注样本来支撑严格结论。

优化后可以这样答：

后续我会建立一批文旅领域标注问题集，每个问题标注标准答案、相关文档 chunk、相关实体和关系。然后分别评估 top-k recall、MRR、nDCG、答案正确率和引用忠实度。

如何优化：

1. 构建人工标注 benchmark。
2. 标注每个问题对应的 gold chunks 和 gold entities。
3. 对比不同检索方案：纯向量、BM25、向量+BM25、GraphRAG。
4. 增加线上日志，用真实问题分析失败案例。

## 四、GraphRAG

### 1. 知识图谱里有哪些实体类型？

当前版本可以这样答：

项目里针对文旅场景设计了比较细的实体类型，主要包括：

1. 景点，比如象鼻山、两江四湖、日月双塔。
2. 地点，比如码头、车站、景区入口等。
3. 交通工具，比如公交线路、游船。
4. 票务/价格，比如成人票、儿童票、优惠票、具体金额。
5. 规则/政策，比如身高限制、购票规则、开放时间规则。
6. 活动/演艺，比如夜游、演出项目。
7. 设施，比如母婴室、停车场、游客中心。
8. 组织/公司，比如旅游公司、景区运营方。

这些类型定义在 Pydantic schema 里，用于约束 LLM 抽取结果。

优化后可以这样答：

后续可以把实体类型进一步规范成更接近业务数据库的 schema，比如把价格、时间、身高条件从普通实体中拆成属性，减少图谱里“210元”这类值节点过多的问题。

如何优化：

1. 区分实体节点和属性值。
2. 把价格、时间、身高限制结构化为属性。
3. 增加实体唯一 ID，避免同名实体冲突。
4. 建立景区、票种、规则、交通站点之间的更严格 schema。

### 2. 关系类型是怎么设计的？

当前版本可以这样答：

关系类型主要围绕文旅问答高频需求设计，包括：

1. `LOCATED_IN`：位于。
2. `NEAR`：毗邻。
3. `COSTS`：价格为。
4. `CONTAINS`：包含。
5. `ROUTE_TO`：通往或到达。
6. `REQUIRES`：要求。
7. `APPLIES_TO`：适用于。
8. `HAS_FACILITY`：拥有设施。

这些关系可以覆盖大部分景区信息查询，比如景点属于哪个区域、票种多少钱、儿童票适用于什么条件、从某站点怎么到码头、景区拥有哪些服务设施。

优化后可以这样答：

后续可以按业务场景继续细化关系，比如 `OPEN_AT`、`CLOSE_AT`、`HAS_TICKET_TYPE`、`DEPARTS_FROM`、`ARRIVES_AT`、`OPERATED_BY`，让路线和票务查询更结构化。

如何优化：

1. 梳理文旅核心业务问题，反推关系 schema。
2. 减少自由关系类型，增加枚举约束。
3. 对路线相关关系增加距离、耗时、交通方式等属性。
4. 对票务关系增加价格、币种、适用人群、有效期等属性。

### 3. 实体和关系是人工抽取还是 LLM 抽取？

当前版本可以这样答：

当前主要是 LLM 抽取。系统把文档切成 chunk 后，针对每个 chunk 构造图谱抽取 prompt，要求 LLM 输出严格 JSON，里面包含 entities 和 relationships。然后用 Pydantic 模型把 JSON 转成结构化对象，再写入 Neo4j。

这种方式的优点是开发效率高，适合原型阶段快速从非结构化文旅资料中构建图谱。缺点是 LLM 抽取存在格式错误、漏抽、错抽和关系不统一的问题，所以项目里也加了一些清洗逻辑，比如删除自环、孤立节点和归一化部分关系类型。

优化后可以这样答：

生产化时我会采用“规则 + LLM + 人工审核”的混合抽取方式。对于票价、时间、公交线路、身高规则等结构化强的信息，用规则或表格解析更可靠；对于景点介绍、特色描述等开放信息，再用 LLM 抽取。

如何优化：

1. 票价、时间、线路用正则或表格解析。
2. 景点、活动、描述类信息用 LLM。
3. 对高价值实体做人工审核或半自动校验。
4. 抽取后做实体对齐和关系归一化。

### 4. LLM 抽取三元组时怎么保证 JSON 格式稳定？

当前版本可以这样答：

当前主要通过 prompt 约束和后处理保证。prompt 中明确要求只输出 JSON，不要 Markdown，不要解释，并给出标准 JSON 示例。返回后会先清理 ```json 这类代码块标记，再用 `json.loads` 解析，最后用 Pydantic schema 校验字段。

如果 JSON 解析失败，当前会跳过该 chunk，避免错误数据写入图谱。

优化后可以这样答：

更好的方案是使用结构化输出能力，比如 LangChain Pydantic parser、OpenAI function calling 或支持 JSON schema 的模型接口。这样可以减少格式错误。同时，对于解析失败的 chunk，可以加入自动修复 prompt，而不是直接跳过。

如何优化：

1. 使用 PydanticOutputParser 或 JSON schema。
2. JSON 解析失败后调用一次 repair chain。
3. 对抽取结果做字段枚举校验。
4. 对异常样本记录日志，后续集中分析。

### 5. 如果抽取出错误实体或脏关系，怎么清洗？

当前版本可以这样答：

当前项目里有基础 DQA 逻辑，主要包括：

1. 删除自环关系，比如实体指向自己。
2. 删除孤立节点，避免图谱里出现没有任何关系的实体。
3. 对部分关系名称做归一化，比如把不同表达统一成 `LOCATED_IN` 或 `NEAR`。
4. 写入实体时使用 `MERGE`，减少完全重复节点。

不过目前清洗还比较基础，对同义实体合并、错别字纠正、关系冲突处理还不完善。

优化后可以这样答：

后续我会增加实体标准化和图谱质量校验。比如通过别名词典、embedding 相似度和编辑距离合并“两江四湖景区”“两江四湖”等同义实体；对价格、时间等冲突信息保留来源和版本，避免直接覆盖。

如何优化：

1. 建立实体 alias 表。
2. 使用规则 + embedding 做实体对齐。
3. 所有关系保留 source chunk 和更新时间。
4. 对冲突价格或规则标记冲突，不直接覆盖。
5. 增加图谱质量统计，比如孤立节点率、重复实体率、关系类型分布。

### 6. Neo4j 里数据 schema 是什么？

当前版本可以这样答：

当前 Neo4j schema 比较简洁。核心节点是 `Entity`，包含 `name`、`type`、`description` 等属性。实体之间通过关系连接，关系类型来自抽取结果，比如 `LOCATED_IN`、`COSTS`、`ROUTE_TO` 等，关系上可以带 `description`。

社区发现后，还会生成 `Community` 节点。`Community` 节点包含 `id`、`level`、`title`、`summary`、`full_content` 等属性，并通过 `HAS_MEMBER` 连接到实体。如果是层次化社区，还会用 `IN_COMMUNITY` 表示社区之间的层级关系。

优化后可以这样答：

后续我会把 schema 拆得更规范，比如增加 `Chunk` 节点，把文档块和实体关联起来；增加 `Document` 节点记录文件来源；对票务、交通、时间规则等高频结构化信息设计专门节点或属性。

如何优化：

1. 增加 `Document` 和 `Chunk` 节点。
2. 建立 `(Entity)-[:MENTIONED_IN]->(Chunk)`。
3. 对 `Ticket`、`Route`、`Policy` 等建立专门类型。
4. 所有实体和关系保留 source、confidence、updated_at。

### 7. Local Graph Search 是怎么做的？

当前版本可以这样答：

Local Search 面向具体事实问题。流程是：

1. 从用户问题中抽取核心实体，比如景点名、票种、地点。
2. 用这些实体到 Neo4j 里查询一跳邻居，拿到相关实体、关系和描述。
3. 同时调用向量/BM25 混合检索，召回相关文档片段。
4. 把图谱上下文和文档上下文一起拼到 prompt 中，让 LLM 基于两类证据回答。

这样做的好处是具体事实可以优先参考结构化图谱，比如票价、路线、适用规则；同时保留原文 chunk 作为补充，避免图谱抽取不完整。

优化后可以这样答：

后续 Local Search 可以从一跳邻居扩展成可控多跳检索，并根据关系类型过滤。例如路线问题重点查 `ROUTE_TO`，票价问题重点查 `COSTS` 和 `APPLIES_TO`。

如何优化：

1. 根据 query intent 选择关系类型。
2. 支持 2-hop 但设置节点数量上限。
3. 对图谱结果按关系类型和实体匹配分数排序。
4. 图谱结果和文本结果统一进入 rerank 或 evidence selector。

### 8. Global Search 是怎么做的？

当前版本可以这样答：

Global Search 面向宏观总结类问题。项目里先从 Neo4j 加载实体关系图，用 Leiden 或 hierarchical Leiden 做社区发现，把相关实体划分成社区。然后针对每个社区收集内部实体和关系，调用 LLM 生成社区标题和摘要，写回 `Community` 节点。

在线查询时，Global Search 会读取已有的社区摘要，按层级组织上下文，让 LLM 基于这些摘要生成全局性回答。这适合回答“整体介绍一下某景区服务体系”“游客主要关心哪些票务政策”“多个景点有什么共同特色”等问题。

优化后可以这样答：

后续可以把 Global Search 做成更标准的 map-reduce。先根据问题筛选相关社区，而不是读取所有社区摘要；再对多个社区摘要做 map 提取和 reduce 汇总，减少无关上下文。

如何优化：

1. 给 Community summary 建 embedding 索引。
2. 查询时先召回相关社区。
3. 对社区摘要做 map-reduce 汇总。
4. 层次化社区优先使用高层摘要，不足时下钻到低层社区。

### 9. 社区发现用了什么算法？Leiden 和 Louvain 区别是什么？

当前版本可以这样答：

项目里主要尝试了 Leiden，也有 Louvain 作为 fallback。社区发现的目的是把图谱中关系紧密的实体聚成社区，再为每个社区生成摘要，供 Global Search 使用。

Louvain 是经典的模块度优化算法，速度快，但可能产生连通性不好的社区。Leiden 可以看作对 Louvain 的改进，它在优化过程中加入了细化阶段，通常能得到连通性和质量更好的社区。

在这个项目里，我使用 graspologic 的 Leiden 或 hierarchical Leiden，把 Neo4j 中的实体关系加载成 NetworkX 图，再运行社区发现，最后把 Community 节点和 HAS_MEMBER 关系写回 Neo4j。

优化后可以这样答：

后续我会结合业务语义对社区做约束。纯图算法只看连接结构，不一定符合旅游业务主题。可以把实体类型、关系类型、文档来源和 embedding 相似度结合起来，提高社区摘要质量。

如何优化：

1. 对不同关系类型设置边权。
2. 把 `NEAR`、`ROUTE_TO`、`CONTAINS` 等关系区别处理。
3. 对社区结果做人工抽样评估。
4. 根据业务主题合并或拆分社区。

### 10. 社区摘要的作用是什么？

当前版本可以这样答：

社区摘要的作用是把一组关系紧密的实体和关系压缩成一段可读的高层知识。图谱原始三元组适合查具体事实，但如果用户问宏观问题，直接把大量三元组塞给 LLM 会很冗长。社区摘要可以提前把这些局部子图总结成主题，比如“两江四湖票务规则”“桂林市区交通到达方式”等。

在线 Global Search 时，系统读取这些摘要，作为宏观上下文生成回答。这样可以降低上下文长度，也让全局问题更容易回答。

优化后可以这样答：

后续社区摘要可以分层维护。底层摘要保留具体事实，高层摘要总结趋势和结构。查询时根据问题粒度决定使用哪一层摘要。

如何优化：

1. 维护 level 0、level 1、level 2 多级摘要。
2. 高层摘要用于宏观问题，底层摘要用于细节追问。
3. 摘要中保留关键实体和来源引用。
4. 文档更新后只增量更新受影响社区。

### 11. GraphRAG 相比普通 RAG 在你项目里带来了什么收益？能举例吗？

当前版本可以这样答：

GraphRAG 的收益主要体现在多实体关联问题上。普通 RAG 只能召回相似文本，而 GraphRAG 可以沿着实体关系补充上下文。

比如用户问“两江四湖夜游成人票和儿童票政策分别是什么，适用于什么人群？”普通向量检索可能只召回票价表的一部分。GraphRAG 可以把“夜游两江四湖成人票”“210元”“儿童票”“身高 1.2-1.5 米”等实体和 `COSTS`、`APPLIES_TO` 关系组织起来，让 LLM 更容易生成结构化答案。

再比如路线类问题，图谱中的 `ROUTE_TO`、`NEAR` 关系可以帮助系统把车站、码头、景点之间的关系补齐。

优化后可以这样答：

更理想的回答应该带有量化结果，比如在一组多景点关联问题上，GraphRAG 相比纯向量 RAG 的正确率提升多少、遗漏率降低多少。目前项目还缺少严格消融实验。

如何优化：

1. 准备多实体、多跳、路线类问题集。
2. 对比纯向量 RAG、向量+BM25、GraphRAG。
3. 统计答案正确率、实体覆盖率、关系覆盖率。
4. 记录典型 case 作为展示材料。

### 12. 如果图谱里没有命中实体，会怎么降级？

当前版本可以这样答：

当前 Local Search 如果图谱没有查到相关实体，会把图谱上下文标记为无相关记录，然后继续使用向量/BM25 混合检索的文档上下文回答。也就是说，图谱不是唯一信息源，图谱缺失时会降级到普通 RAG。

另外，部分脚本里还有 fuzzy_search，会用 `CONTAINS` 做模糊匹配，尽量解决用户输入和图谱实体名称不完全一致的问题。

优化后可以这样答：

后续可以做更强的实体链接和降级策略。比如先用 alias 表、拼音/编辑距离、embedding 相似度找候选实体；如果仍然找不到，再走纯文档检索；如果文档检索也低置信度，就明确回复资料不足。

如何优化：

1. 建立实体别名库。
2. 实体识别后做候选实体召回和重排。
3. 支持模糊匹配、同义词匹配和 embedding 匹配。
4. 图谱失败时自动 fallback 到 hybrid RAG。
5. fallback 结果低置信时拒答。

## 五、路线规划与文旅业务

### 1. 你说支持旅游规划，具体规划到了什么程度？

当前版本可以这样答：

当前项目支持的是基于文旅资料和图谱关系的行程规划辅助，而不是严格意义上的路径优化系统。它可以根据景点信息、地理邻近关系、票务规则、交通说明和文档中的推荐信息，为用户生成多天游玩建议。

比如用户问“三天游玩象鼻山、两江四湖、漓江怎么安排”，系统会先检索这些景点的介绍、交通和关联信息，再由 LLM 汇总成一个较合理的行程建议。

但当前还没有完整实现旅行商问题、最短路径、时间窗约束、开放时间约束、实时交通等优化算法。所以我在面试中会把它表述为“旅游规划辅助决策”，而不是“严格自动路线优化”。

优化后可以这样答：

优化后可以把它升级成约束规划系统：把景点作为节点，交通耗时作为边权，开放时间、游玩时长、预算、用户偏好作为约束，再用路径搜索或启发式算法生成更可解释的路线。

如何优化：

1. 为每个景点补充经纬度、开放时间、建议游玩时长。
2. 为交通边补充距离、耗时、交通方式。
3. 引入地图 API 或离线距离矩阵。
4. 使用贪心、动态规划、遗传算法或 OR-Tools 做路线优化。
5. LLM 只负责解释和润色，不直接决定路线。

### 2. 系统是否真的做路径最短路、时间约束、开放时间约束？

当前版本可以这样答：

当前版本没有完整实现这些严格约束。系统目前更多是通过检索景区资料和图谱关系，让 LLM 生成行程建议。图谱里可以表达部分路线关系和邻近关系，但还没有把它们转成可计算的时间距离网络，也没有接入开放时间和实时交通的约束求解。

所以如果面试官问到，我会明确说：当前系统是“基于检索增强的规划建议”，不是“严格路径优化引擎”。

优化后可以这样答：

如果要实现真正路线规划，我会把文旅知识图谱和路径规划模块拆开：图谱负责提供景点、票价、规则、交通说明；路径规划模块负责基于结构化数据计算最优路线。

如何优化：

1. 将 `ROUTE_TO` 关系增加 `duration`、`distance`、`transport_mode` 属性。
2. 为景点增加 `open_time`、`close_time`、`visit_duration`。
3. 根据用户天数和偏好建立约束优化模型。
4. 先生成候选路线，再用 LLM 解释路线选择理由。

### 3. 如果用户问“三天游桂林怎么安排”，你是怎么生成路线的？

当前版本可以这样答：

当前系统会把这个问题当成复合型任务处理。Planner 会把它拆成多个子任务，比如查询核心景点信息、查询景点之间的交通或关联关系、获取整体旅游建议。然后 local_search 会检索具体景点和路线信息，global_search 可以补充宏观旅游资源或整体建议。最后 Reporter 把检索结果组织成按天安排的回答。

这个路线安排主要基于检索到的资料和 LLM 的规划能力，比如把邻近景点放在同一天，把夜游项目安排在晚上，把耗时较长的景点单独安排。

但当前没有严格计算每两个景点之间的距离和交通时间，因此生成的是合理建议，不是数学意义上的最优路线。

优化后可以这样答：

优化后我会让系统先结构化提取用户约束，比如出发地、天数、偏好、预算、同行人群，再用路线规划模块生成候选行程，最后用 GraphRAG 补充每个景点的票务、交通和注意事项。

如何优化：

1. 先做用户约束抽取。
2. 构建景点候选池。
3. 根据距离、开放时间和游玩时长生成路线。
4. 用 GraphRAG 补充每个景点的说明。
5. 输出时标明“上午/下午/晚上”和交通方式。

### 4. 多景点之间的关联关系来自哪里？

当前版本可以这样答：

当前多景点关联主要来自两部分。

第一部分来自文档中显式描述的信息。比如文档里提到某景区包含哪些子景点、某码头靠近哪些景点、某公交线路通往哪里，系统会通过 LLM 抽取成 `CONTAINS`、`NEAR`、`ROUTE_TO` 等关系。

第二部分来自社区发现。图谱中连接较紧密的实体会被聚成社区，社区摘要可以帮助回答跨景点、跨主题的整体问题。

但当前项目还没有接入真实地图坐标和距离 API，所以“邻近关系”主要依赖文档描述和抽取结果。

优化后可以这样答：

后续可以把文档抽取关系和地理空间关系结合起来。文档提供语义关系，地图 API 或经纬度计算提供真实距离关系，两者结合会更可靠。

如何优化：

1. 给景点和站点补充经纬度。
2. 根据地理距离自动生成 `NEAR` 候选关系。
3. 用地图路线 API 生成交通耗时。
4. 对 LLM 抽取的邻近关系进行地理校验。

### 5. 票价、交通、开放时间这类信息如何保证准确？

当前版本可以这样答：

当前主要通过两点保证准确性：

1. 回答必须基于检索到的文档和图谱上下文，不允许 LLM 自己编造。
2. 结构化信息会尽量进入 Neo4j，比如票价、适用条件、路线说明等，在回答时优先使用图谱中的结构化事实。

但是准确性的上限仍然取决于原始资料是否准确、是否过期，以及 LLM 抽取是否正确。当前项目还没有接入实时官网或票务系统，所以不能保证实时票价和开放时间。

优化后可以这样答：

生产系统中，我会对票价、开放时间、交通这类高风险信息引入来源、更新时间和置信度，并优先从官方结构化数据源同步，而不是完全依赖 LLM 抽取。

如何优化：

1. 每条票价和开放时间记录保留 source 和 updated_at。
2. 高风险信息优先使用官方 API 或人工维护表。
3. 回答中显示“资料来源”和“更新时间”。
4. 对不同来源冲突的信息做冲突提示。
5. 对实时信息明确提示系统知识边界。

### 6. 如果文档里有过期信息，系统怎么处理？

当前版本可以这样答：

当前版本没有完整的时效性管理。文档解析后进入向量库和图谱，系统默认这些资料是可信的。如果原始文档过期，回答也可能基于过期信息。

因此我会在面试中明确说明：当前系统适合固定知识库下的问答和规划辅助，不适合直接回答强实时问题。

优化后可以这样答：

后续需要建立数据版本和更新机制。每个文档、chunk、实体和关系都带上来源和更新时间。回答涉及票价、开放时间、政策时，系统优先选择最新版本，并在答案中提示资料时间。

如何优化：

1. 文档入库时记录 `source_file`、`version`、`updated_at`。
2. 图谱关系和 chunk 绑定来源。
3. 新文档导入时做增量更新和旧数据失效。
4. 对高时效问题触发外部数据源查询或拒答。

### 7. 如果用户问实时天气、实时票价、实时交通，你的系统能回答吗？

当前版本可以这样答：

当前系统不能可靠回答实时天气、实时票价和实时交通。它的知识主要来自本地文旅资料、向量库和 Neo4j 图谱，不包含实时数据源。

如果用户问这类问题，比较稳妥的回答应该是：当前知识库不包含实时信息，建议以景区官网、票务平台或地图交通信息为准。系统可以回答资料中已有的历史或静态规则，但不能保证实时准确。

优化后可以这样答：

如果要支持实时问题，需要接入外部工具，比如天气 API、景区票务 API、地图交通 API。多智能体架构本身适合扩展这些工具：Planner 识别实时意图，Executor 路由到实时 API 工具，Reporter 再把实时结果和本地知识库结合起来回答。

如何优化：

1. 增加 weather tool、ticket tool、map traffic tool。
2. Planner 增加实时意图识别。
3. 对实时工具结果设置更高优先级。
4. 回答中标明实时数据获取时间。
5. 实时 API 失败时明确降级，不用旧知识冒充实时答案。

## 六、缓存设计

### 1. 你为什么要做缓存？

当前版本可以这样答：

因为这个项目的在线链路比较长，一个复杂问题可能会触发 Planner 拆解、多个检索工具、rerank、Reporter 汇总等多次 LLM 调用和模型推理。如果用户重复问相同问题，或者问语义非常接近的问题，每次都完整跑一遍成本比较高，延迟也比较长。

所以我实现了缓存模块，用来复用已经生成过的结果。缓存主要解决两个问题：

1. 降低重复推理成本，减少本地 LLM 和 reranker 的重复调用。
2. 提升响应速度，尤其是多智能体流程中相似问题重复出现时。

项目里的缓存支持 memory、disk 和 hybrid 后端，也支持语义相似命中，用于处理“表述不同但语义相近”的问题。

优化后可以这样答：

后续我会把缓存分成多层：检索结果缓存、rerank 结果缓存、最终回答缓存和工具调用缓存。不同层的 TTL 和失效策略不同，避免把最终回答缓存当成所有场景的唯一缓存。

如何优化：

1. 区分 query cache、retrieval cache、rerank cache、answer cache。
2. 对票价、开放时间等高时效内容设置较短 TTL。
3. 对通用景区介绍设置较长 TTL。
4. 增加缓存命中日志和统计面板。

### 2. 缓存的 key 是怎么生成的？

当前版本可以这样答：

项目里使用策略模式生成缓存 key。基础策略是 SimpleCacheStrategy，主要根据 query 文本生成 key。对于带上下文的场景，会使用 ContextAwareCacheStrategy，把 query 和上下文参数一起纳入 key，比如模型、角色、temperature 等信息。

这样做是为了避免不同模型、不同 Agent 角色、不同上下文下的结果互相污染。例如同一个 query，如果一个是在 Planner 角色下生成任务图，另一个是在 Reporter 角色下生成最终回答，它们不能共用同一个缓存结果。

优化后可以这样答：

后续我会把 key 设计得更显式，比如统一包含 `query_hash`、`model_id`、`agent_role`、`tool_name`、`knowledge_base_version`、`prompt_version`。这样当知识库或 prompt 更新时，可以自动避免旧缓存误用。

如何优化：

1. 引入知识库版本号。
2. 引入 prompt 版本号。
3. 引入工具名称和 Agent 角色。
4. 对 key 生成逻辑做单元测试。
5. 缓存 metadata 中保留 key 构成，方便排查误命中。

### 3. 什么是 Context-Aware 缓存？

当前版本可以这样答：

Context-Aware 缓存就是缓存 key 不只看用户问题本身，还看问题发生的上下文。因为在多智能体系统里，同一句话可能出现在不同阶段，含义不同。

比如“总结以上内容”，如果出现在 Reporter 阶段，它依赖前面检索到的证据；如果出现在普通问答阶段，它可能是用户要求总结景区资料。只用 query 做 key，就可能把不同上下文下的结果混用。

所以项目里在检测到模型、Agent 角色、temperature 等上下文参数时，会切换到上下文感知策略，把这些信息一起参与 key 生成。

优化后可以这样答：

更完整的 Context-Aware 缓存应该把会话 ID、任务 ID、工具名、知识库版本、prompt 版本和关键上下文摘要都纳入缓存条件。这样可以更精确地区分“同问法不同上下文”的情况。

如何优化：

1. key 中加入 `thread_id` 或 `session_id`。
2. key 中加入 `task_type` 和 `tool_name`。
3. key 中加入 evidence hash，防止同问题不同证据误命中。
4. 对最终回答缓存和中间工具缓存使用不同策略。

### 4. 如何避免 A 用户的问题命中 B 用户上下文里的缓存？

当前版本可以这样答：

当前缓存设计里，session cache 可以通过 thread_id 或上下文参数区分不同会话；global cache 则用于可以跨会话复用的通用问题。也就是说，涉及会话上下文的问题应该走 session 级缓存，而不是全局缓存。

不过当前实现仍然是原型级，如果开发者没有正确传入 thread_id 或上下文参数，仍然可能出现过度复用的问题。因此我不会说它已经完全解决了所有缓存污染，只能说项目已经实现了上下文感知缓存的基本机制。

优化后可以这样答：

生产环境中，我会把缓存分成私有缓存和公共缓存。默认用户上下文相关问题只进私有缓存，只有经过判断的静态知识问答才进入公共缓存。

如何优化：

1. 默认所有带 session 的请求使用 session cache。
2. 只有无用户隐私、无上下文依赖的问答进入 global cache。
3. key 中强制包含 user/session scope。
4. 对 global cache 写入增加白名单策略。
5. 增加缓存命中审计日志。

### 5. 语义相似缓存命中是怎么实现的？

当前版本可以这样答：

项目里在缓存写入时，如果开启了向量匹配，会对 query 生成 embedding，并把 key 到 embedding 的映射保存在内存索引里。读取缓存时，如果精确 key 没命中，就会对当前 query 生成 embedding，和已有缓存 query 的 embedding 做相似度计算。如果最高相似度超过阈值，就认为可以复用对应缓存。

这个机制解决的是同义问题复用，比如“如何安排三天桂林行程”和“桂林三日游怎么规划”。

优化后可以这样答：

后续我会让语义缓存命中更谨慎。只对低风险的解释类、总结类问题开启语义复用；对票价、时间、政策、实时类问题提高阈值或关闭语义复用。

如何优化：

1. 按任务类型设置不同 similarity threshold。
2. 对高风险事实类问题只允许精确命中。
3. 语义命中后进行一次轻量校验，确认问题约束一致。
4. 记录命中 query 和原 query，便于审计。

### 6. 相似度阈值怎么选？

当前版本可以这样答：

当前阈值主要是经验设置。简单任务可以设置得低一些，比如 0.85，提高复用率；复杂或高风险任务设置得高一些，比如 0.95，降低误命中风险。项目里 `CacheConfig.for_agent` 对 simple 和 hard 难度做了不同配置。

优化后可以这样答：

更合理的阈值应该通过验证集调出来。准备一批 query pair，标注哪些可以复用、哪些不能复用，然后画出不同阈值下的 precision 和 recall，选择一个误命中风险可接受的阈值。

如何优化：

1. 构造相似问题对和非相似问题对。
2. 统计不同阈值下的误命中率。
3. 对不同任务类型分别调阈值。
4. 上线后根据用户反馈动态调整。

### 7. 模糊命中会不会导致错误复用？怎么防止缓存污染？

当前版本可以这样答：

会有这个风险。比如“成人票多少钱”和“儿童票多少钱”语义很接近，但答案不能复用。如果语义缓存只看 embedding，相似度可能很高，从而误命中。

当前项目通过提高复杂任务阈值、区分上下文策略来降低风险，但还不是完全充分。

优化后可以这样答：

防止缓存污染需要多层机制。除了 embedding 相似度，还要比较关键实体、问题类型、约束条件和时间敏感性。只有这些都一致时才允许复用。

如何优化：

1. 缓存 metadata 中保存抽取出的实体和意图。
2. 命中前比较实体集合，比如成人票和儿童票不能互相命中。
3. 对价格、时间、政策类 query 关闭模糊命中。
4. 引入缓存质量分数，用户负反馈后降低或删除缓存。
5. 定期清理低质量缓存。

### 8. 缓存支持哪些后端？memory、disk、hybrid 分别怎么工作？

当前版本可以这样答：

项目里的缓存后端包括 memory、disk 和 hybrid。

memory backend 把缓存放在内存里，速度快，但进程重启后丢失，适合临时缓存。

disk backend 把缓存序列化到磁盘，速度比内存慢，但可以持久化，适合复用历史结果。

hybrid backend 结合两者，优先查内存，内存没有再查磁盘；写入时同时维护内存和磁盘。这样可以兼顾速度和持久化。

外层还包了一层 ThreadSafeCacheBackend，用锁保证多线程访问时的安全性。

优化后可以这样答：

生产环境中可以增加 Redis 作为分布式缓存后端，解决多进程、多实例部署时本地 memory cache 不共享的问题。

如何优化：

1. 增加 Redis backend。
2. 支持按 namespace 区分不同业务缓存。
3. 支持缓存容量淘汰策略，比如 LRU。
4. 对 disk cache 增加压缩和定期清理。

### 9. 线程安全怎么保证？

当前版本可以这样答：

缓存后端外面有 ThreadSafeCacheBackend，它用锁包裹 get、set、delete、clear 等操作，避免多线程同时读写缓存结构导致数据不一致。

另外，server/utils 里还有一个 ChatConcurrentManager，用 session_id 维度维护锁，避免同一个会话同时进入多个处理流程，造成上下文混乱。

优化后可以这样答：

如果部署为 Web 服务，除了线程锁，还要考虑多进程和多实例。单机线程锁只能保证当前进程内安全，不能保证多个服务实例之间一致。

如何优化：

1. 单进程内继续使用线程锁。
2. 多进程部署时使用 Redis 分布式锁或队列。
3. 对同一 session 请求做串行化处理。
4. 对耗时任务引入任务队列。

### 10. 缓存命中后会不会跳过必要的检索更新？

当前版本可以这样答：

有这个可能。如果知识库已经更新，但缓存还没失效，命中缓存就会返回旧答案。当前项目主要通过 TTL 控制缓存过期，但还没有和知识库版本强绑定。

优化后可以这样答：

生产化时，缓存 key 必须包含 knowledge_base_version 或 index_version。只要文档、向量索引或图谱更新，版本号变化，旧缓存自然不会命中。

如何优化：

1. 给知识库维护版本号。
2. 每次索引构建或图谱更新后刷新版本。
3. 缓存 key 中加入版本号。
4. 对实时性数据设置更短 TTL 或不缓存。

## 八、评估与效果

### 1. 你怎么证明 GraphRAG 比普通 RAG 好？

当前版本可以这样答：

当前项目已经实现了普通 RAG、混合检索和 GraphRAG 的链路，也有评估模块，但还没有形成严格的消融实验报告。所以我不能夸大说已经用完整数据证明了 GraphRAG 一定更好。

从机制上看，GraphRAG 对多实体关联问题更有优势，比如票价规则、交通路线、多景点关系这类问题。普通 RAG 只召回文本片段，GraphRAG 可以额外提供实体和关系上下文。

优化后可以这样答：

我会构建一组测试集，把问题分成单点事实、多实体关联、路线规划、宏观总结几类。然后比较普通向量 RAG、向量+BM25、GraphRAG 三种方案在答案正确率、实体覆盖率、关系覆盖率和引用忠实度上的表现。

如何优化：

1. 构建标注 benchmark。
2. 做 ablation study。
3. 记录每类问题的效果差异。
4. 展示典型 case，而不是只给主观判断。

### 2. 你做了哪些评估指标？

当前版本可以这样答：

项目里的 evaluation 模块主要包含几类指标：

1. 答案质量指标：Exact Match、F1，以及 LLM fallback 语义评分。
2. 检索质量指标：retrieval precision、retrieval utilization、chunk utilization。
3. 性能指标：retrieval latency。
4. 缓存指标：cache hit rate、cache average time saved。

这些指标用于观察系统回答是否接近标准答案、检索证据是否被使用、延迟表现如何，以及缓存是否减少重复计算。

优化后可以这样答：

后续应该增加 RAG 更常用的指标，比如 faithfulness、answer relevance、context precision、context recall，同时引入人工评估。

如何优化：

1. 增加上下文召回率。
2. 增加答案忠实度评估。
3. 增加人工打分维度。
4. 将评估结果自动保存成报告。

### 3. EM、F1 对中文开放式问答是否合适？

当前版本可以这样答：

EM 和 F1 对开放式中文问答不是特别合适。EM 要求答案完全匹配，过于严格；F1 基于分词重叠，对同义表达和结构化回答不够友好。

所以项目里除了规则 EM/F1，也加入了 LLM fallback 评分，用 LLM 判断两个答案在语义上是否等价。不过 LLM 评分本身也有不稳定性，所以只能作为辅助指标。

优化后可以这样答：

更好的评估方式应该是结合规则、语义模型和人工评估。对于票价、时间这类事实问题，可以用规则抽取关键字段精确比对；对于规划类开放答案，可以用人工或 LLM 按覆盖度、合理性、忠实度打分。

如何优化：

1. 对结构化事实做字段级评估。
2. 对开放规划做 rubric 评分。
3. 使用人工评估校验 LLM 评估。
4. 避免只看 EM/F1 得出结论。

### 4. 检索 precision 和 utilization 怎么定义？

当前版本可以这样答：

retrieval precision 主要关注最终回答中引用或使用的实体，有多少确实来自检索结果。它用于判断回答是否被检索证据支撑，减少幻觉。

retrieval utilization 关注检索出来的实体或片段有多少被最终回答真正使用。它用于判断检索是否召回了大量无用信息。

简单说，precision 看“答案里的内容是否来自检索”，utilization 看“检索到的内容是否被答案利用”。

优化后可以这样答：

后续我会把这两个指标细化到 chunk 和三元组级别。比如答案中每个关键事实都要能映射到某个 chunk 或某条图谱关系。

如何优化：

1. 答案生成时要求引用 evidence_id。
2. 对每个关键句做来源匹配。
3. 统计 chunk-level precision/recall。
4. 统计 entity/relation coverage。

### 5. 有没有人工标注数据集？

当前版本可以这样答：

当前项目还没有完整的人工标注 benchmark。数据主要来自文旅资料和一些测试问题，评估模块已经搭好，但标注集还不够系统。

这是一个短板。如果要严谨证明效果，需要人工标注问题、标准答案、相关 chunk、相关实体和关系。

优化后可以这样答：

我会构建一个小规模但高质量的文旅评估集，比如 100 到 300 个问题，覆盖景点查询、票价政策、交通路线、多景点规划和宏观总结。

如何优化：

1. 收集真实游客问题。
2. 人工标注标准答案。
3. 标注相关文档片段和图谱实体关系。
4. 每类问题保持数量均衡。

### 6. 有没有做 ablation study，比如只用向量、向量+BM25、GraphRAG？

当前版本可以这样答：

当前代码层面支持这些链路，比如普通 RAG、混合检索、GraphRAG 都有实现，但还没有形成完整的 ablation study 报告。

优化后可以这样答：

后续会固定同一批测试问题，分别跑：

1. 向量 RAG。
2. BM25。
3. 向量 + BM25 + RRF。
4. 向量 + BM25 + rerank。
5. GraphRAG + 混合检索。

然后比较答案正确率、检索召回率、延迟和成本。

如何优化：

1. 封装统一 evaluation runner。
2. 每种检索策略实现同一接口。
3. 自动输出对比表。
4. 保存失败案例。

### 7. 缓存带来了多少响应时间下降？

当前版本可以这样答：

项目里有缓存测试脚本，会比较第一次运行和第二次命中缓存的耗时，也有 cache performance metric 用于记录命中率和节省时间。但当前还没有整理成正式的压测报告。

从机制上说，缓存命中后可以跳过 Planner、检索和 Reporter 的部分推理流程，延迟会明显降低，尤其是复杂多智能体问题。

优化后可以这样答：

后续我会做标准化压测，统计 P50、P95、P99 延迟，分别对比冷启动、无缓存、精确命中、语义命中几种情况。

如何优化：

1. 固定测试问题集。
2. 记录每次请求各阶段耗时。
3. 输出缓存命中率和平均节省时间。
4. 区分精确命中和语义命中。

### 8. 并发能力有没有压测？QPS 大概多少？

当前版本可以这样答：

当前仓库有一个 `stress_test.py`，用于测试 OpenAI-compatible LLM 接口的吞吐、token 生成速度和最大并发，但它更偏模型服务压测，不是完整应用链路压测。

项目里也有线程安全缓存和会话级并发锁，但我不会说已经完成了生产级高并发压测。更准确的说法是：系统考虑了并发访问下的缓存安全和会话隔离，但完整 QPS 还需要进一步压测。

优化后可以这样答：

生产化前需要对完整链路做压测，包括 API 层、检索层、rerank、LLM、Neo4j 和缓存。

如何优化：

1. 补 FastAPI 服务后用 Locust 或 k6 压测。
2. 统计 P50/P95/P99。
3. 分别压测冷缓存和热缓存。
4. 找出瓶颈是 LLM、rerank、Neo4j 还是 FAISS。

### 9. 失败案例有哪些？

当前版本可以这样答：

主要失败案例有几类：

1. LLM 抽取图谱时 JSON 格式错误，导致某些 chunk 被跳过。
2. 实体识别不准，导致图谱检索没有命中。
3. 文档资料过期，导致回答基于旧信息。
4. Planner 任务拆解不稳定，可能把具体问题路由成 global_search。
5. 缓存语义命中如果阈值不合适，可能错误复用相似但约束不同的问题。

优化后可以这样答：

我会针对这些失败案例做监控和 fallback。比如抽取失败自动修复，实体未命中走混合检索，Planner 路由错误时触发二次检索，缓存命中前比较实体和意图。

如何优化：

1. 建立失败案例库。
2. 增加日志和 trace。
3. 对失败 case 做回归测试。
4. 逐步优化 prompt、schema 和检索策略。

### 10. 你觉得目前效果最不稳定的环节在哪里？

当前版本可以这样答：

最不稳定的是 LLM 参与结构化输出的环节，包括 Planner 输出任务 JSON、图谱抽取输出 JSON、社区摘要输出 JSON。这些环节如果模型输出格式不稳定，会影响后续流程。

其次是实体链接和路由判断。文旅实体名称比较多，如果用户说法和图谱名称不一致，图谱检索可能命中失败。

优化后可以这样答：

我会优先把结构化输出和实体链接做稳。结构化输出用 schema 约束，实体链接用 alias、规则和 embedding 候选召回。

如何优化：

1. 使用 JSON schema 或 function calling。
2. 增加 JSON repair。
3. 建实体别名库。
4. 路由加规则兜底。
5. 对每个阶段增加可观测日志。

## 九、LLM 与 Prompt

### 1. 为什么选择 `qwen2.5:14b`？

当前版本可以这样答：

选择 `qwen2.5:14b` 主要是因为项目希望本地运行，避免依赖外部大模型 API，同时中文能力要比较稳定。14B 规模在本地部署和回答质量之间相对平衡，适合做中文文旅问答、任务拆解和报告生成。

不过这不是唯一选择。项目里也有部分脚本尝试过 Kimi/OpenAI-compatible 接口，说明模型层可以替换。

优化后可以这样答：

后续我会把模型选择配置化，并对不同任务选择不同模型。比如 Planner 用较稳定的结构化输出模型，生成回答用中文表达更强的模型，embedding 和 rerank 使用专门模型。

如何优化：

1. 模型名称配置化。
2. 不同 Agent 支持不同模型。
3. 对模型做效果和延迟评估。
4. 支持本地模型和云端模型切换。

### 2. 本地 LLM 和云端 LLM 的取舍是什么？

当前版本可以这样答：

本地 LLM 的优点是数据不出本地，成本可控，适合处理内部文旅资料和原型验证。缺点是推理速度和效果受本地硬件影响，模型能力也可能弱于云端大模型。

云端 LLM 的优点是能力强、结构化输出更稳定、维护成本低；缺点是有调用成本、网络依赖和数据合规问题。

这个项目当前偏本地化，所以主要用 Ollama + qwen2.5。

优化后可以这样答：

后续可以做混合策略：默认用本地模型处理普通问答，复杂结构化输出或高质量报告生成时可切换到云端模型。

如何优化：

1. 建统一 LLM provider 接口。
2. 支持 local/cloud fallback。
3. 根据任务复杂度路由模型。
4. 对敏感数据默认不出本地。

### 3. Prompt 里如何约束模型不要编造？

当前版本可以这样答：

项目里的生成 prompt 会明确要求模型严格基于上下文回答，如果上下文中没有答案，要回答不知道或资料未提及。HybridAgent 的系统 prompt 也强调必须坚守事实，优先使用检索上下文，不能凭常识猜测。

GraphRAG 回答时，还会把图谱上下文和文档上下文分开标记，让模型知道结构化图谱信息优先级更高。

优化后可以这样答：

仅靠 prompt 不能完全防幻觉。后续应该加入引用约束和答案后校验，要求每个关键事实都能对应到 evidence。

如何优化：

1. 答案中强制输出引用。
2. 生成后检查引用是否存在。
3. 无引用事实删除或要求重答。
4. 对低置信检索结果触发拒答。

### 4. Planner 输出 JSON 不合法怎么办？

当前版本可以这样答：

当前实现会先从 LLM 输出中提取 JSON 代码块，如果没有代码块，就截取第一个 `{` 到最后一个 `}`。解析失败时会记录错误，并生成一个默认 local_search 任务，保证流程不中断。

优化后可以这样答：

后续可以使用更严格的结构化输出，比如 Pydantic parser 或 function calling。解析失败时不要只兜底，还可以让 LLM 根据错误信息修复一次 JSON。

如何优化：

1. 使用 PydanticOutputParser。
2. 增加 JSON repair prompt。
3. 对字段枚举做校验。
4. 超过修复次数后才 fallback。

### 5. 抽取图谱时 LLM 漏实体怎么办？

当前版本可以这样答：

当前没有完全解决漏抽问题。系统主要依赖 prompt 让 LLM 抽取关键实体和关系。如果漏抽，图谱就不完整，但在线查询时仍然可以通过向量/BM25 文档检索补充。

优化后可以这样答：

后续可以结合规则抽取和多轮抽取减少漏抽。比如先用规则抽取价格、时间、公交线路，再让 LLM 抽取景点、活动和关系。还可以对同一 chunk 做实体抽取和关系抽取两阶段处理。

如何优化：

1. 规则抽取结构化强的信息。
2. LLM 抽取语义关系。
3. 对重要文档做二次抽取。
4. 把抽取结果和原文 chunk 关联，便于补全。

### 6. 你有没有用结构化输出、function calling 或 Pydantic parser？

当前版本可以这样答：

当前使用了 Pydantic 模型定义实体、关系、任务图和状态对象，但 LLM 输出解析主要还是 prompt + JSON 后处理，还没有完整使用 function calling 或严格 Pydantic parser。

也就是说，Pydantic 主要用于解析后的数据校验，不是模型端强约束输出。

优化后可以这样答：

后续会把 Planner、图谱抽取、社区摘要这些结构化输出环节都迁移到 Pydantic parser 或 function calling。

如何优化：

1. 给 TaskGraph、GraphResult、CommunitySummary 定义 parser。
2. Prompt 中注入格式说明。
3. 解析失败自动 retry。
4. 模型支持时使用 JSON schema response format。

### 7. temperature 设置多少？为什么？

当前版本可以这样答：

Agent 编排和工具路由类任务更适合低 temperature，项目里的 ChatOllama 工厂里设置了 temperature=0，目的是让 Planner、Router、Reporter 的输出更稳定，尤其是 JSON 和工具调用场景。

对于最终回答生成，也倾向较低 temperature，因为文旅问答需要事实准确，不需要太强创造性。

优化后可以这样答：

可以按任务类型设置 temperature。结构化任务用 0，开放式文案润色可以稍高，但涉及事实的回答仍然保持低温。

如何优化：

1. Planner temperature=0。
2. 图谱抽取 temperature=0。
3. Reporter 可设置 0 到 0.2。
4. 所有温度参数配置化。

### 8. 多轮对话上下文怎么管理？

当前版本可以这样答：

`graphrag_agent` 里基于 LangGraph 使用 MemorySaver 保存对话状态，并通过 thread_id 区分会话。缓存层也支持 thread_id，用于会话级缓存。

但 `agent_system` 的 Plan-Execute-Report 入口更偏单轮复杂问题处理，多轮上下文管理还不是重点。

优化后可以这样答：

后续我会把多轮上下文分为短期对话记忆和长期用户偏好。短期记忆用于理解追问，长期偏好用于规划时考虑用户喜好，比如亲子游、预算、节奏偏慢等。

如何优化：

1. session 级对话历史持久化。
2. 对历史做摘要，避免上下文过长。
3. 用户偏好结构化存储。
4. 多轮追问时做 query rewrite，把省略信息补全。

## 十、最容易被追问的风险点

### 1. 你说“高并发”，具体并发模型是什么？有压测数据吗？

当前版本可以这样答：

我会谨慎表述。当前项目实现了线程安全缓存和基于 session_id 的会话锁，也有模型接口压测脚本，但还没有完整 Web 服务链路的高并发压测数据。因此我不会说已经达到生产级高并发。

更准确的说法是：项目在缓存读写和会话处理上考虑了并发安全，为后续服务化和压测打了基础。

优化后可以这样答：

补充 FastAPI 服务后，我会用 Locust 或 k6 对完整链路做压测，统计 QPS、P95 延迟、错误率、缓存命中率和瓶颈阶段。

如何优化：

1. 完成 API 服务化。
2. 增加异步请求处理。
3. 用 Redis 做共享缓存。
4. 对 LLM 调用做队列和限流。

### 2. 你说“防污染缓存”，能举一个污染场景和你的解决方法吗？

当前版本可以这样答：

一个典型污染场景是：用户 A 问“儿童票多少钱”，用户 B 问“成人票多少钱”，两个问题语义很接近，如果只用向量相似度做缓存命中，可能把儿童票答案返回给成人票问题。

当前项目通过 Context-Aware key 和较高相似度阈值降低这个风险，但还没有完全解决所有业务约束误命中。

优化后可以这样答：

更完整的方案是缓存 metadata 里保存实体、意图和约束条件。命中缓存前不仅比较语义相似度，还要比较实体集合和关键槽位，比如票种、景点、时间、人群必须一致。

如何优化：

1. query 入缓存前抽取 slots。
2. 命中时比较 slots。
3. 票务/政策类问题只允许严格匹配。
4. 用户负反馈后降低缓存质量分。

### 3. 你说“旅游规划”，是不是只是基于检索结果生成建议，而不是严格路径优化？

当前版本可以这样答：

是的，当前更准确地说是“旅游规划辅助建议”，不是严格路径优化。系统可以基于文旅资料、景点关系、票务和交通信息生成合理行程，但还没有实现开放时间、交通耗时、距离矩阵等约束下的优化算法。

优化后可以这样答：

如果要做到严格规划，需要引入结构化景点数据和路径优化模块，让 LLM 负责解释，不负责核心路径计算。

如何优化：

1. 补充经纬度和开放时间。
2. 构建交通耗时图。
3. 使用 OR-Tools 或启发式算法规划路线。
4. LLM 只做自然语言生成。

### 4. 你说“多智能体”，是不是只是几个模块顺序调用？

当前版本可以这样答：

项目里的多智能体是偏工程编排式的，不是开放式多 Agent 对话。它的价值在于把复杂任务拆成 Planner、Worker、Reporter 三个角色，每个角色有明确输入输出和边界。

当前主要是顺序执行，但任务图里有 depends_on 和 execution_mode，为后续并行和更复杂 DAG 调度留了空间。

优化后可以这样答：

后续可以把 Worker 扩展成多个专用 Agent，比如票务 Agent、路线 Agent、景点 Agent、总结 Agent，并支持并行执行和反思重规划。

如何优化：

1. 增加更多专用工具。
2. 支持 parallel execution。
3. 实现 reflection 和 re-plan。
4. 增加任务级 trace 和可视化。

### 5. 你说“GraphRAG”，图谱检索到底参与了最终回答多少？

当前版本可以这样答：

在 Local Search 中，图谱上下文会和向量文档上下文一起进入 prompt。具体事实类问题会优先使用图谱结构化信息，比如票价、适用规则、路线关系。但如果图谱没有命中，系统会降级使用向量/BM25 文档检索。

所以图谱不是替代文本 RAG，而是作为结构化增强信息参与最终回答。

优化后可以这样答：

后续可以在答案中标注哪些事实来自图谱、哪些来自文档，从而量化图谱贡献。

如何优化：

1. 图谱 evidence 和文本 evidence 分别编号。
2. 答案引用标明来源类型。
3. 统计答案中图谱事实占比。
4. 对比有无图谱的回答差异。

### 6. 你说“自动化 Pipeline”，是否能一键从原始文件到可查询系统？

当前版本可以这样答：

当前是流程上具备这些步骤，但还不是严格意义的一键 Pipeline。项目里有文档解析、切块、向量化、图谱构建脚本，但入口分散，需要按步骤运行。

优化后可以这样答：

后续我会封装成统一命令，比如 `python manage.py build --source data/`，自动完成解析、索引构建、图谱构建和摘要生成。

如何优化：

1. 整合离线构建脚本。
2. 增加任务进度和失败重试。
3. 构建结果持久化。
4. 写 README 指导一键构建。

### 7. 你说“FastAPI”，项目里接口在哪？

当前版本可以这样答：

当前仓库没有完整 FastAPI 接口，所以我不应该把 FastAPI 写成已完成技术栈。如果简历中出现 FastAPI，需要么补齐服务层，要么删掉，避免面试追问时对不上。

优化后可以这样答：

补齐后我会提供问答、规划、文档导入、索引构建和健康检查接口。

如何优化：

2. 封装服务层。

### 8. 你有没有把 API key 写进代码？如果有，线上怎么处理？

当前版本可以这样答：

当前有一个脚本里硬编码过 Kimi API key，这是不规范的，也不应该进入线上环境。正确做法是立即废弃这个 key，并迁移到环境变量或密钥管理系统。

优化后可以这样答：

线上环境所有密钥都通过环境变量、配置中心或 secret manager 注入，代码库只保留 `.env.example`，不保存真实 key。

2. 轮换已经暴露的 key。
3. 增加 `.gitignore` 忽略 `.env`。
4. 使用 secret scanning 防止再次提交。

## 十一、我最可能深入追的 5 个问题

### 1. 请画一下你的系统架构图，并讲清楚在线查询链路。

当前版本可以这样答：

我会把架构分成四层：

1. 数据层：DOCX、PDF、Markdown、JSON 等文旅资料。
2. 索引和图谱层：FAISS 向量库、BM25 索引、Neo4j 知识图谱、Community 摘要。
3. 检索和工具层：local_search、global_search、hybrid retrieval、rerank。
4. Agent 编排层：Planner、Worker、Reporter、Cache。

在线查询链路是：用户问题进入 Orchestrator，Planner 拆成任务图，Worker 根据任务类型调用 local 或 global search，工具返回图谱和文本证据，Reporter 汇总生成最终回答，最后写入缓存。

优化后可以这样答：

如果要面试展示，我建议你画成左右结构：左边离线构建，右边在线问答。这样面试官很容易理解。

如何优化表达：

1. 先讲离线：文档到索引和图谱。
2. 再讲在线：问题到任务、检索、汇总。
3. 最后讲缓存和评估作为横切能力。

### 2. 普通 RAG 回答不了的一个具体问题是什么？GraphRAG 怎么解决？

当前版本可以这样答：

比如用户问：“两江四湖夜游成人票和儿童票分别多少钱，儿童票适用于什么条件，附近还有哪些景点可以一起安排？”

普通 RAG 可能只召回票价表，或者只召回景点介绍，信息容易不完整。GraphRAG 会先识别“两江四湖”“夜游票”“儿童票”等实体，再从 Neo4j 查 `COSTS`、`APPLIES_TO`、`NEAR` 等关系，把票价、适用条件和邻近景点一起补充到上下文中。

优化后可以这样答：

最好准备一个真实 case，对比纯向量 RAG 和 GraphRAG 的回答差异，展示 GraphRAG 多召回了哪些实体关系。

如何优化：

1. 选 3 个典型多跳问题。
2. 保存两种方案回答。
3. 标出 GraphRAG 补充的关系。
4. 做成面试讲解材料。

### 3. 你的混合检索：向量、BM25、RRF、rerank，每一步输入输出是什么？

当前版本可以这样答：

输入是用户 query 或改写后的 query。

向量检索输入 query，输出 top-k Document 和相似度分数。

BM25 输入 query 的 jieba 分词结果，输出 top-k Document 和 BM25 分数。

RRF 输入两路排序结果，按排名融合，输出去重后的候选 Document 列表。

rerank 输入 query 和候选 Document pair，输出相关性分数，排序后取 top-k。

最终 top-k Document 被拼成上下文，交给 LLM 生成回答。

优化后可以这样答：

后续每一步都应该保留 trace，包括召回来源、原始分数、融合分数、rerank 分数，便于调试为什么某个答案错了。

如何优化：

1. 定义统一 RetrievalResult 对象。
2. 保存 scores 和 source。
3. 日志输出每阶段 top-k。
4. 在评估系统中分析召回失败原因。

### 4. 缓存的 Context-Aware 具体怎么避免误命中？

当前版本可以这样答：

当前做法是 key 不只依赖 query，还会考虑模型、角色、temperature 等上下文参数。当这些上下文存在时，缓存管理器会选择 ContextAwareCacheStrategy，从而避免同一个 query 在不同 Agent 阶段共用同一个结果。

但当前还没有把所有业务槽位都纳入 key，比如票种、人群、时间等。因此它是基础防污染机制，不是完整最终形态。

优化后可以这样答：

更完整的方式是“上下文 key + 业务槽位校验 + 语义阈值”。即便语义相似度高，也必须满足实体、意图、约束一致才能命中。

如何优化：

1. key 加 session、role、tool、kb_version。
2. metadata 保存 entities、intent、slots。
3. 命中前做 slot comparison。
4. 高风险问题关闭模糊命中。
