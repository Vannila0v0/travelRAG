# 项目改进：从 Demo 到可评测、可迭代系统

## 一、整体改进思路

这个项目最开始已经可以完成基础的文旅问答和路线规划，但我没有只停留在“能跑通 Demo”的层面，而是进一步做了系统化评测和针对性优化。

RAG、GraphRAG 和 Multi-Agent 项目很容易出现一个问题：表面上能生成答案，但很难判断答案是否稳定、检索链路是否正确、来源是否可追溯、复杂任务是否真的经过了有效拆解。

所以我后续专门构建了一套小型 benchmark，用来验证系统在真实文旅场景下的表现，并基于评测结果反向优化系统。

这次改进的核心目标是：

1. 让系统不只是能回答，而是能被量化评测。
2. 让问题能被定位到具体模块，比如路由、检索、GraphRAG、Agent 汇总。
3. 让每次优化都有指标对比，而不是只靠人工主观判断。

## 二、问题发现

我先把文旅问答场景拆成几类典型问题：

1. 单景点事实查询  
   例如门票、价格、开放时间、交通方式。

2. 规则政策类查询  
   例如儿童票、退票、免票、半价票、乘坐限制。

3. 多景点关联查询  
   例如某个景区有哪些特色景点，不同景点之间有什么关系。

4. 全局推荐与路线规划  
   例如桂林市区一日游、适合夜游的项目、跨景点路线安排。

5. 多智能体复杂规划  
   例如“帮我规划一天桂林市区路线，包含交通和票价”。

在评测过程中，我发现系统存在几个隐性问题：

1. 自动路由不稳定，部分规则类问题被错误路由到纯向量检索。
2. Agent 最终答案能生成，但没有把 Worker 阶段检索到的 sources 透传出来，导致复杂规划结果不可追溯。
3. Global GraphRAG 能做宏观总结，但有些答案覆盖不完整，例如夜游推荐没有稳定覆盖“游船”和“桂林千古情”。
4. 只靠人工观察答案质量不可靠，因此需要一套可重复运行的评测机制。

2. “漓江精华游船儿童免票和半价票规则是什么？”也被路由到了 `vector`。
3. “遇龙河竹筏乘坐有什么规则？”也应该走 `hybrid`。
4. “银子岩溶洞有哪些特色景点？”被误判成了 `global`，但它其实是具体景点详情查询。

Agent 单独评测也暴露了问题：

```text
answer_keyword_recall = 100%
source_recall = 0%
has_source_rate = 0%
```

也就是说，Agent 的最终报告内容看起来是对的，但没有把检索来源传出来。这个在工程上是不合格的，因为复杂规划结果不可追溯。

## 五、问题一：自动路由优化

原来的自动路由主要依赖比较粗的关键词判断。例如只要出现“有哪些”，就容易被判断成 `global`。但实际上，“银子岩有哪些特色景点？”是一个具体景点详情问题，不应该走全局总结。

因此我将路由规则调整为更明确的优先级：

```text
票务 / 规则 / 政策 / 交通类问题 -> hybrid
规划 / 路线 / 行程类问题 -> global
桂林 / 市区级推荐汇总类问题 -> global
具体景点的特色景点 / 设施 / 景观类问题 -> hybrid
普通介绍类问题 -> vector
```

具体来说：

1. 包含 `退票`、`免票`、`半价`、`儿童票`、`乘坐规则`、`怎么去`、`交通`、`票价` 等强规则词的问题，优先路由到 `hybrid`。
2. 包含 `规划`、`路线`、`一日游`、`怎么玩`、`怎么安排` 等跨景点规划词的问题，路由到 `global`。
3. 类似“桂林有哪些适合夜游的项目”这种城市级推荐问题，路由到 `global`。
4. 类似“银子岩有哪些特色景点”这种具体景点细节问题，路由到 `hybrid`。

优化后重新跑非 Agent 评测：

```text
route_accuracy: 63.64% -> 100.00%
answer_keyword_recall: 88.64% -> 93.18%
source_recall: 59.09% -> 72.73%
```

这个结果说明，路由修正不仅让链路选择更准确，也提升了答案和来源质量。

## 六、问题二：Agent Sources 透传

第二个问题是多智能体最终报告没有来源。

我分析后发现，Worker 子任务其实通过工具调用拿到了检索结果，但执行记录里没有完整保存 sources，Orchestrator 也没有把所有子任务 sources 汇总到最终状态里，最后 `QueryEngine` 返回 Agent 结果时 sources 为空。

因此我改造了 Agent 的证据链路：

1. `tool_registry` 保留工具返回的完整 sources。
2. `retrieval_executor` 把每个子任务的 `route` 和 `sources` 写入 `ExecutionRecord`。
3. `ExecutionRecord` 增加 `route` 和 `sources` 字段。
4. `PlanExecuteState` 增加全局 `sources` 字段。
5. `Orchestrator` 在 Worker 执行结束后统一收集所有子任务 sources，并按 `chunk_id` / `doc_id` 去重。
6. `QueryEngine.ask(..., route="agent")` 最终把 `state.sources` 转成标准 `QueryResult.sources` 返回。

优化后单独跑 Agent 评测：

```text
修改前：
source_recall = 0%
has_source_rate = 0%

修改后：
source_recall = 100%
has_source_rate = 100%
```

这说明 Agent 不再只是“生成一个看起来合理的规划报告”，而是每个子任务的检索证据都能被保留下来，最终结果具备可追溯性。

当前还可以继续优化的方向包括：

## 十、Global GraphRAG 来源回溯优化

在前两轮优化之后，系统还剩下一个比较典型的问题：Global Search 的答案依赖 community summary，但返回 sources 时主要返回 community 本身，没有稳定回溯到底层原始文档和 chunk。

这个问题会影响 GraphRAG 的可解释性。因为用户看到的是宏观总结，但系统需要说明这个总结具体来自哪些原始文档、哪些切片。

因此我对 Global Search 做了第三轮优化。

### 1. 问题表现

典型失败样本是：

```text
问题：桂林有哪些适合夜游的项目？
期望路由：global
实际路由：global
问题：答案和来源没有稳定覆盖“两江四湖游船”和“桂林千古情”
```

这说明路由本身是正确的，问题出在 global 检索阶段：

1. community summary 能提供宏观信息，但覆盖不一定完整。
2. sources 返回 community 本身，缺少原始文档来源。
3. 对“夜游”这类场景词，没有扩展到“演出、游船、灯光、千古情、两江四湖”等相关概念。

### 2. 优化方式

我将 Global Search 改造成两层返回：

```text
第一层：社区级结果
- community_id
- community_title
- community_summary
- score

第二层：证据级来源
- doc_id
- chunk_id
- file_name
- source_path
- 原始 chunk 文本
```

查询链路从原来的：

```text
query -> community summary -> LLM answer -> community source
```

改成：

```text
query -> community summary
      -> Community - HAS_MEMBER -> Entity - MENTIONED_IN -> Chunk -> Document
      -> community summary + supporting chunks
      -> LLM answer
      -> community source + original chunk sources
```

同时我增加了 query expansion 和场景权重：

1. `夜游` 扩展到 `晚上`、`夜间`、`演出`、`游船`、`灯光`、`千古情`、`两江四湖`。
2. `一日游/路线` 扩展到 `路线`、`行程`、`上午`、`下午`、`晚上`、`交通`、`票价`。
3. 对市区路线问题，提高“桂林市区旅游与水域管理社区”和“两江四湖景区夜游与票务体系”的召回权重。
4. 对夜游问题，提高“桂林千古情”和“两江四湖”相关社区的召回权重。

### 3. 优化结果

针对两个 Global 样本单独评测：

### 4. 面试表述

面试中可以这样讲：

```text
我后续发现 Global GraphRAG 还有一个可追溯性问题：系统能基于 community summary 生成宏观答案，但 sources 主要停留在社区摘要层，没有稳定返回原始文档证据。

所以我把 Global Search 改成了两层检索。第一层召回 community summary，用来提供宏观结构；第二层沿着 Community -> Entity -> Chunk -> Document 的路径，找到支撑该社区摘要的原始文档切片。最终 LLM 同时看到社区摘要和原文片段，而接口返回时也同时返回 community source 和底层 chunk source。
```

### 5. 为什么不返回社区下的所有原始切片

这里还涉及一个重要设计点：一个社区的原始支撑切片理论上会很多。

因为社区是由底层实体节点构成的，而每个实体又通过 `MENTIONED_IN` 关系关联到原始 chunk。因此严格来说，一个社区的完整支撑来源应该是：

```text
Community
  -> HAS_MEMBER
  -> Entity
  -> MENTIONED_IN
  -> Chunk
  -> Document
```

也就是说，社区底下所有成员实体关联到的 chunk，都可以被认为是这个社区的原始支撑证据集合。

但在线查询时不能把这些 chunk 全部塞给 LLM，也不应该全部返回给用户，原因包括：

1. 数量可能很多  
   一个 community 可能有几十个实体，每个实体又可能出现在多个 chunk 中，展开后可能达到几十到几百个 chunk。

2. 重复度很高  
   同一个 chunk 可能同时提到多个实体，所以会被多次命中。

3. 和当前问题不一定都相关  
   例如“两江四湖票务体系”社区里可能同时包含成人票、儿童票、退票、码头、航线、演出等内容。如果用户只问“夜游项目”，就没有必要把所有退票规则和儿童票规则都返回。

因此我在设计上区分了两层概念：

```text
社区完整支撑集：
这个社区底下所有实体关联过的全部 chunk。
用于离线追溯、审计、调试、重建 summary。

查询时证据子集：
从完整支撑集中按当前 query 重新打分，选 top-k 个最相关 chunk。
用于在线回答问题和返回 sources。
```

当前实现采用的是第二种，也就是 query-time supporting chunks：

```text
先召回 community
再从该 community 的所有关联 chunk 中，按 query token 命中、实体命中、mention_count 等信号重新排序
每个 community 只取前几个 chunk
```

所以系统返回的 sources 不是“这个社区的全部原始来源”，而是：

```text
该社区中与当前问题最相关的原始支撑来源。
```

面试中可以这样表述：

```text
一个 community 理论上会关联大量原始 chunk，我没有在在线查询时全部返回，而是把它分成完整支撑集和查询支撑子集。完整支撑集可以用于离线审计和 summary 重建；在线回答时会先召回 community summary，再沿 Community -> Entity -> Chunk 找到候选 chunks，并根据当前 query 做二次排序，只取 top-k 作为回答证据。这样既保证了可追溯性，又避免上下文过长和噪声过多。
```

如果进一步生产化，还可以在建图或生成社区摘要时，给 `Community` 节点预先保存：

```text
supporting_chunk_ids
supporting_doc_ids
top_entity_names
summary_source_refs
```

这样查询时不需要每次实时展开所有成员实体，而是可以先读取这些预计算字段，再做 query-time rerank。

## 十一、Agent DAG 并发调度优化

在准确性和可追溯性优化完成后，我继续观察 Agent 复杂规划任务的耗时问题。

典型样本是：

```text
问题：帮我规划一天桂林市区游玩路线，包含交通和票价
路由：agent
```

在完成 Agent sources 透传后，这个样本虽然质量指标已经达标：

```text
route_accuracy = 100.00%
answer_keyword_recall = 100.00%
source_recall = 100.00%
has_source_rate = 100.00%
```

但耗时较高：

```text
avg_latency_seconds = 185.15
```

### 1. 问题原因

原来的 `WorkerCoordinator` 是串行执行任务的。

但 Planner 生成的是一个 DAG，而不是简单列表。很多子任务之间并没有依赖关系，例如：

```text
task_001: global_search，获取桂林市区核心景点概览

task_002: local_search，查询象鼻山信息
task_003: local_search，查询靖江王府信息
task_004: local_search，查询两江四湖信息

task_005: reflection，综合前面结果进行路线校验
```

其中 `task_002`、`task_003`、`task_004` 都只依赖 `task_001`，彼此之间没有依赖关系。原先串行执行会导致这些独立检索任务一个接一个调用 LLM 和检索工具，整体耗时被累加。

### 2. 优化方式

我将 Worker 调度器改成基于 DAG 的 ready batch 并发执行：

```text
每一轮获取依赖已满足的 ready tasks
如果 ready tasks 只有 1 个，则直接执行
如果 ready tasks 有多个，则使用 ThreadPoolExecutor 并发执行
每个任务完成后再更新 task.status、execution_records、completed_task_ids
下一轮继续检查新的 ready tasks
```

并发数通过环境变量控制：

```text
AGENT_MAX_WORKERS=3
```

默认值为 3，避免一次性打太多 LLM 请求。

同时我补了两个线程安全点：

1. `tool_registry.get_query_engine()` 增加锁，避免并发下重复创建 `QueryEngine` 单例。
2. `VectorQueryEngine.vectorstore` 的 FAISS 懒加载增加锁，避免多个 Worker 同时首次加载索引。

### 3. 优化结果

优化后重新跑 Agent q012：

```text
route_accuracy = 100.00%
answer_keyword_recall = 100.00%
source_recall = 100.00%
has_source_rate = 100.00%
avg_latency_seconds = 124.64
```

和优化前对比：

```text
Agent 耗时：185.15s -> 124.64s
下降约：32.7%
```

日志中也能看到调度器按 DAG 执行：

```text
task_001 单独执行
task_002 / task_003 / task_004 并发执行
task_005 等前置任务完成后再执行
```

这说明 Agent 不再是简单串行跑工具，而是能够根据任务依赖关系做并发调度。

### 4. 面试表述

面试中可以这样讲：

```text
我后续对多智能体执行阶段做了性能优化。Planner 生成的任务本质上是一个 DAG，而不是普通列表。早期 Worker 是串行执行的，即使多个子任务之间没有依赖，也会一个接一个调用检索和 LLM，导致复杂规划耗时较高。

所以我把 WorkerCoordinator 改成了基于 ready tasks 的批次调度。每一轮先找出依赖已经满足的任务，如果有多个 ready tasks，就用 ThreadPoolExecutor 并发执行；如果后续任务依赖这些结果，比如 reflection，就等前置任务完成后再执行。

同时我给 QueryEngine 单例和 FAISS 向量索引懒加载加了锁，避免并发初始化问题。优化后，在 q012 复杂规划样本上，Agent 的 source_recall 和答案召回仍保持 100%，耗时从约 185 秒降到了约 125 秒，下降约 32.7%。
```

## 十二、Agent 工具缓存接入与复盘

在并发调度之后，我继续尝试把缓存接入到 Agent Worker 的工具调用层。

### 1. 为什么缓存放在工具层

多智能体系统和普通单轮 RAG 不同。普通 RAG 如果只缓存最终用户问题，命中率通常不高，因为用户的问题表述变化很大。

但 Agent 会把复杂任务拆成多个子任务，例如：

```text
查询象鼻山开放时间和门票
查询靖江王府交通方式
查询两江四湖票价和码头
查询桂林市区核心景点概览
```

这些子任务在不同用户的路线规划问题中更容易重复。因此缓存放在 Worker 工具调用层，比只缓存最终答案更符合多智能体系统的复用粒度。

### 2. 当前实现方式

我在 `ToolAdapter.structured_search()` 中接入了确定性工具缓存。

缓存 value 不是只存 answer，而是保存完整工具结果：

```json
{
  "success": true,
  "answer": "...",
  "data": {
    "route": "hybrid",
    "sources": [],
    "metadata": {
      "cache_hit": false,
      "cache_layer": "agent_tool",
      "cache_schema_version": "tool-cache-v1"
    }
  }
}
```

这样缓存命中时仍然可以回放 sources，不会破坏前面做好的可追溯链路。

### 3. Key 生成策略

工具缓存不走 simple key，而是强制使用 context-aware key。

缓存签名包含：

```text
cache_schema_version
route
task_type
agent_role
entities
intent_tags
model
temperature
prompt_version
data_version
```

其中：

```text
route：vector / hybrid / global
task_type：local_search / global_search / reflection
entities：Planner 抽取出的实体，经过轻量归一化
intent_tags：根据任务描述抽取的意图标签，如 overview、ticket、time、traffic、food、route_check
```

这样做的目的不是提高答案能力，而是避免缓存污染。例如同一句“两江四湖票价”，如果分别走 `vector`、`hybrid`、`global`，不能共用同一个缓存。

### 4. 实际评测结果

我用 q012 复杂 Agent 样本做了两轮测试：

```text
第一次：预热缓存
第二次：复用缓存
```

但在实际测试中，第二次没有稳定命中：

```text
warmup:
tool_cache.hits = 0
tool_cache.misses = 5

hit run:
tool_cache.hits = 0
tool_cache.misses = 5
```

原因不是缓存读写失效，而是 Planner 每次生成的任务拆解差异较大。

例如第一次可能拆成：

```text
分别查询象鼻山、靖江王府、两江四湖
```

第二次可能拆成：

```text
全局查询多个景点
单独查询交通
单独查询餐饮
两个 reflection 任务
```

在这种情况下，确定性工具缓存会认为它们是不同任务，因此不会命中。

### 5. 复盘结论

这次实现说明了一个关键点：

```text
工具缓存已经具备工程接入能力，但在 Planner 输出不稳定时，单纯的确定性工具缓存收益有限。
```

这和前面的判断是一致的：工具缓存不是核心智能能力，而是成本和延迟优化手段。它只有在子任务稳定、重复率较高的情况下才有明显收益。

如果要让它在 Agent 场景中真正稳定生效，后续需要做两件事之一：

1. 缓存 Planner 计划  
   对相同或高度相似的复杂问题，先复用稳定任务图，让 Worker 子任务更稳定。

2. 做受限语义匹配  
   不是全局语义缓存，而是在 `route`、`task_type`、`entities`、`data_version` 一致或高度接近的前提下，再用 embedding 相似度判断是否复用。

### 6. 面试表述

面试中可以这样讲：

```text
我尝试把缓存接入到 Agent Worker 的工具调用层，缓存的是完整工具结果，包括 answer、route、sources 和 metadata。Key 不是简单 query hash，而是包含 route、task_type、entities、intent_tags、model、temperature、prompt version、data version 等上下文，避免不同检索链路和不同智能体角色之间污染。

但测试后我发现，确定性工具缓存的收益受 Planner 稳定性影响很大。同一个用户问题，Planner 第二次可能拆成不同的任务图，导致工具层 key 不一致，从而无法命中。因此我不会把这部分夸大成核心创新，它更像是一个可选的工程优化点。

如果要继续提升它的实际收益，下一步应该先让 Planner 输出更稳定，或者缓存 Planner 计划；再考虑受限语义匹配，而不是直接做全局模糊缓存。
```

每条样本包含以下字段：

```json
{
  "id": "q001",
  "question": "用户问题",
  "expected_route": "期望路由",
  "expected_answer_keywords": ["答案中应该覆盖的关键词"],
  "expected_sources": ["期望命中的文档或社区来源"],
  "tags": ["问题类型标签"]
}
```

这个 benchmark 主要评测以下指标：

1. `route_accuracy`  
   自动路由是否走到了正确链路，例如 `vector`、`hybrid`、`global`、`agent`。

2. `answer_keyword_recall`  
   答案是否覆盖关键事实。

3. `source_recall`  
   返回的来源是否命中预期文档或社区。

4. `has_source_rate`  
   系统是否返回了可追溯来源。

5. `latency_seconds`  
   响应耗时，用来观察性能成本。

这套评测不是只看“答案像不像”，而是重点验证三个问题：

```text
有没有走对检索链路？
有没有答到关键事实？
有没有可追溯证据？
```

## 十三、Planner 任务规范化层

### 1. 为什么要做任务规范化

在多智能体架构里，Planner 的职责是把用户问题拆成一个任务 DAG，后续 Worker 再按任务类型调用不同工具，Reporter 最后汇总结果。

但实际测试时发现，同一个问题多次调用 Planner，LLM 生成的任务数量、任务命名、任务描述和依赖关系可能会有差异。例如同样是：

```text
帮我规划一天桂林市区游玩路线，包含交通和票价
```

Planner 可能一次拆成“景点查询、交通查询、票价查询、路线整合”，另一次拆成“景点概览、详细门票、交通方式、综合规划”。这些拆法语义上接近，但结构不稳定。

这会带来三个问题：

```text
1. Worker 执行不稳定：同一类问题每次执行的任务 DAG 可能不同。
2. 工具缓存难以复用：缓存 key 依赖 task_type、entities、intent_tags，如果 Planner 输出不稳定，缓存很难命中。
3. 评测和排查困难：同一个 benchmark 样本每次走的执行路径不同，不利于定位问题。
```

所以这里新增了一个任务规范化层，把“LLM 生成的自由任务图”转换成“系统可稳定执行的标准任务图”。

### 2. 任务规范化层放在哪里

它位于 Planner 和 Worker 之间：

```text
用户问题
  -> IntentRouter 判断是否需要 agent
  -> Planner 生成原始 TaskGraph
  -> TaskNormalizer 规范化 TaskGraph
  -> WorkerCoordinator 按 DAG 执行任务
  -> Reporter 汇总最终答案
```

也就是说，Planner 仍然负责理解用户问题和生成初始计划，但不会把原始计划直接交给 Worker，而是先经过 `TaskNormalizer` 做结构收敛。

### 3. 当前版本怎么实现

当前实现了两类规范化策略。

第一类是针对高频复杂场景的模板化规范化。比如“桂林市区一日游 + 交通 + 票价”这类问题，会被规范成稳定的 5 个任务：

```text
task_001: global_search
获取桂林市区核心景点和一日游路线框架

task_002: local_search
查询象鼻山、靖江王府、两江四湖、日月双塔等景点的票价、开放时间和游玩信息

task_003: local_search
查询桂林市区核心景点之间的交通方式

task_004: local_search
查询桂林市区餐饮补充信息

task_005: reflection
综合景点、交通、票价和餐饮信息，生成一日游路线
```

依赖关系是：

```text
task_001
  -> task_002 / task_003 / task_004
  -> task_005
```

这样 Worker 可以先执行全局规划，再并发执行景点、交通、餐饮三个检索任务，最后由 reflection 任务做综合。

第二类是通用规范化。对于没有命中特定模板的问题，不强行改写业务含义，只做结构清洗：

```text
1. 将任务 ID 统一重编号为 task_001、task_002 ...
2. 修正依赖关系，过滤不存在的 depends_on。
3. 对 entities 做去重和清洗。
4. 统一任务初始状态为 pending。
5. 保留原始 task_type 和 description。
```

这样既不会过度干预 Planner，又能保证后续执行层看到的是一个更规整的任务图。

### 4. 验证结果

新增了 Planner 稳定性测试脚本：

```text
evaluation/planner_stability.py
```

测试方式是对同一个复杂规划问题连续运行 Planner 多次，记录规范化后的任务签名，观察是否稳定。

本次验证结果：

```text
runs = 3
stable = true
unique_signatures = 1
```

说明虽然原始 LLM 输出仍然有差异，但经过规范化层之后，交给 Worker 的任务 DAG 已经稳定。

同时重新跑了 q012 多智能体端到端评测：

```text
route_accuracy = 100.00%
answer_keyword_recall = 100.00%
source_recall = 100.00%
has_source_rate = 100.00%
avg_latency_seconds = 111.35
```

这说明任务规范化没有降低答案质量，反而让执行路径更稳定，并且在当前样本上比之前的并发 Worker 版本耗时进一步下降。

### 5. 面试表述

面试中可以这样讲：

```text
我后续发现，多智能体系统里 Planner 的输出不能完全直接交给执行层，因为 LLM 生成的任务拆解存在随机性。同一个旅游规划问题，它可能每次拆出的任务数量、任务描述、依赖关系都略有不同。语义上可能都对，但对工程系统来说会造成执行路径不稳定，也会影响工具缓存、评测复现和问题排查。

所以我在 Planner 和 Worker 之间加了一层 TaskNormalizer。Planner 仍然负责理解用户问题并生成原始任务图，TaskNormalizer 负责把这个自由形式的任务图收敛成系统可执行的标准 DAG。

对于高频复杂场景，比如“桂林市区一日游，包含交通和票价”，我会把它规范成固定的五类任务：全局路线框架、景点票价开放时间查询、交通查询、餐饮补充查询、最终综合规划。这样 Worker 就可以按照 DAG 先跑 global_search，再并发跑几个 local_search，最后跑 reflection 汇总。

对于没有命中特定模板的问题，我不会强行模板化，只做通用结构清洗，比如任务 ID 重编号、依赖关系校验、实体去重和状态初始化。

这个优化解决的是多智能体从 demo 到工程化时非常关键的一个问题：不能只让 LLM 自由发挥，还要把 LLM 的输出变成稳定、可观测、可评测、可缓存的中间表示。后续我也用一个 planner_stability 脚本验证了这一点，同一个复杂规划问题连续跑 3 次，原始 Planner 输出不同，但规范化后的任务签名只有 1 种，端到端 q012 样本的答案召回和来源召回仍然保持 100%。
```

## 十四、任务模板配置化改造

### 1. 为什么继续改造

上一阶段已经加入了 `TaskNormalizer`，解决了 Planner 输出不稳定的问题。但第一版仍然有一个工程化不足：高频复杂任务的标准 DAG 是写死在代码里的。

这意味着如果后续要支持更多场景，例如“阳朔一日游”“桂林两日游”“亲子游”“雨天备选路线”，就需要不断修改 Python 代码。这样不利于扩展，也不利于把业务模板交给非核心代码开发者维护。

所以这一阶段的目标不是重新做任务规范化层，而是把它从“代码硬编码模板”升级成“配置驱动模板”。

### 2. 当前实现方式

新增配置文件：

```text
configs/task_templates.json
```

模板里定义三类信息：

```text
1. match：模板命中规则，例如必须同时命中地域、规划意图、票价交通等关键词组。
2. execution_mode：任务图执行模式，例如 parallel。
3. nodes：标准任务 DAG，包括 task_id、task_type、description、priority、depends_on、entities。
```

新增模板匹配器：

```text
agent_system/planner/task_template_matcher.py
```

它负责读取 `configs/task_templates.json`，根据用户 query 匹配模板，并将模板转换成 `TaskGraph`。

`TaskNormalizer` 的职责也被收敛了：

```text
Planner 原始 TaskGraph
  -> TaskTemplateMatcher 尝试命中配置模板
  -> 命中：返回配置里的标准 DAG
  -> 未命中：执行通用规范化，包括 task_id 重编号、依赖清洗、entities 去重
```

这样 `TaskNormalizer` 不再关心具体业务模板细节，而是只负责规范化流程。

### 3. 工程兜底

为了避免配置错误影响主流程，我给模板构建加了兜底逻辑：

```text
如果配置文件不存在：记录 warning，回退通用规范化。
如果模板格式错误：跳过该模板，继续匹配后续模板。
如果模板命中但构图失败：记录 warning，回退通用规范化。
```

这样模板系统不会成为 Planner 的单点故障。

### 4. 验证结果

先做了不调用 LLM 的直接检查：

```text
输入：帮我规划一天桂林市区游玩路线，包含交通和票价
输出：parallel 模式，5 个标准任务节点
task_001: global_search
task_002: local_search
task_003: local_search
task_004: local_search
task_005: reflection
```

同时验证了未命中模板时仍然会走通用规范化，例如 task_id 重编号和 entities 去重。

随后重新运行 Planner 稳定性评测：

```text
runs = 3
stable = true
unique_signatures = 1
```

说明即使 Planner 三次原始输出仍然不同，配置化模板仍然可以把最终任务结构收敛成同一种标准 DAG。

端到端 q012 评测这次没有完成，原因不是代码问题，而是当前环境的 Neo4j 没有启动，`localhost:7687` 连接被拒绝；尝试通过 Docker Compose 启动时，当前会话无法连接 Docker daemon，提升权限审批也没有返回。因此本阶段完成了编译检查、模板直接检查和 Planner 稳定性检查，但没有完成依赖 Neo4j 的端到端回归。

### 5. 面试表述

面试中可以这样讲：

```text
第一版任务规范化层已经能解决 Planner 输出不稳定的问题，但当时高频复杂场景的 DAG 还是写死在代码里的。这个设计可以验证思路，但扩展性一般，因为每新增一个旅游业务场景都要改 Python 代码。

所以我后续把任务规范化升级成了配置化任务模板系统。我新增了 task_templates.json，用配置描述模板命中规则、执行模式和标准任务 DAG；再新增 TaskTemplateMatcher，负责读取配置、匹配用户 query，并构造成 TaskGraph。

这样 Planner 仍然可以自由理解用户问题，但执行层看到的是稳定的中间表示。对于命中模板的高频复杂场景，系统直接使用标准 DAG；对于没有命中模板的问题，仍然保留通用规范化逻辑，不会过度模板化。

这个改造的价值是把多智能体系统从“靠 prompt 临场生成任务”推进到“LLM 理解 + 配置化流程约束”的模式。它提升了可维护性、可扩展性和评测稳定性，也为后续支持更多旅游规划场景打下基础。
```

### 6. 规范化的具体收益

任务规范化的价值不只是让任务 ID 更整齐，而是让多智能体执行链路从“LLM 每次自由生成”变成“高频场景可稳定复现”。

第一，执行路径更稳定。

同一个问题：

```text
帮我规划一天桂林市区路线，包含交通和票价
```

Planner 第一次可能拆成：

```text
景点概览 -> 门票查询 -> 交通查询 -> 总结
```

第二次可能拆成：

```text
核心景点查询 -> 象鼻山查询 -> 两江四湖查询 -> 预算汇总
```

这些拆法语义上可能都合理，但系统执行路径不稳定。规范化之后，只要命中模板，就会固定成同一套 DAG：

```text
全局概览 -> 景点票价 / 交通 / 餐饮 -> 综合规划
```

这样更容易调试、复现和评测。

第二，便于并发执行。

标准 DAG 能明确表达任务依赖关系。例如：

```text
task_002 景点票价查询
task_003 交通查询
task_004 餐饮查询
```

它们都只依赖 `task_001`，彼此之间没有依赖，所以 Worker 可以并发执行。如果完全依赖 Planner 每次自由生成依赖关系，Worker 很难稳定判断哪些任务可以并发。

第三，便于缓存复用。

工具缓存的 key 会参考：

```text
task_type
entities
intent_tags
route
```

如果 Planner 每次拆出的子任务都不一样，缓存很难命中。规范化之后，同类问题会产生更稳定的子任务，例如：

```text
local_search + 象鼻山/靖江王府/两江四湖 + ticket/time
```

这样工具层缓存才有更明确的复用基础。

第四，便于问题排查。

如果一个复杂规划问题答错了，可以沿着标准任务链路定位问题：

```text
是 task_001 全局规划错了？
是 task_002 景点票价没查到？
是 task_003 交通信息不足？
还是 task_005 综合规划有问题？
```

如果每次 Planner 输出的任务结构都不一样，问题复现和定位都会困难很多。

### 7. 通用规范化和 Planner 自由发挥的关系

通用规范化不是完全等于“没有规范化”，但它确实保留了 Planner 的自由度。

可以这样理解：

```text
没有规范化：
Planner 输出什么，Worker 就直接执行什么。

通用规范化：
Planner 决定任务含义，但系统会清洗任务结构后再执行。
```

也就是说，通用规范化不会强行改变 Planner 的业务拆解。例如 Planner 生成：

```text
查象鼻山门票
查靖江王府交通
最后总结
```

通用规范化不会把它改成：

```text
查两江四湖
查餐饮
查预算
```

它只做工程层面的整理：

```text
1. task_id 统一为 task_001、task_002 ...
2. depends_on 清理，删除不存在的依赖。
3. entities 去重。
4. description 去掉前后空格。
5. status 初始化为 pending。
```

所以三种层次的区别是：

```text
完全无规范化：
LLM 直接控制执行图，最灵活，但最不稳定。

通用规范化：
LLM 仍然决定任务怎么拆，但系统保证任务图格式干净、依赖合法、ID 稳定。

模板规范化：
对于高频复杂场景，系统直接收敛成标准 DAG，保证执行路径稳定。
```

面试中如果被问到“通用规范化是不是还是让 Planner 自由发挥”，可以这样回答：

```text
是的，通用规范化保留了 Planner 在业务拆解上的自由度，但不是完全放任。它不会改变任务语义，只会约束任务图的工程结构，例如 ID、依赖、实体和状态。对于普通问题，这样可以避免过度模板化；对于高频复杂问题，则通过配置模板进一步收敛成标准 DAG。
```

## 十六、复杂 Agent 测试集扩充

### 1. 为什么要扩充

之前复杂多智能体评测主要依赖 q012：

```text
帮我规划一天桂林市区游玩路线，包含交通和票价
```

这个样本可以验证桂林市区一日游链路，但覆盖面仍然偏窄。为了避免多智能体能力只在单一样本上成立，我新增了一个独立的复杂 Agent 测试集。

新增文件：

```text
evaluation/datasets/agent_planning_qa.jsonl
```

这份测试集不混入原来的主 benchmark，而是专门用于评估复杂旅游规划场景。

### 2. 新增样本覆盖范围

当前新增 6 条复杂 Agent 样本：

```text
a001: 阳朔一日游，包含遇龙河、西街、交通方式和门票预算
a002: 桂林夜游，包含两江四湖、桂林千古情、时间、票价和交通比较
a003: 预算有限的桂林市区一日游，控制门票和交通成本
a004: 桂林亲子一日游，要求轻松，并说明儿童票和交通注意事项
a005: 龙胜温泉一日游，从龙胜县城汽车总站出发，包含交通、门票优惠和游玩重点
a006: 桂林雨天备选路线，尽量选择室内或交通方便的景点，并说明票价
```

每条样本都包含：

```text
id
question
expected_route = agent
expected_answer_keywords
expected_sources
tags
```

这样它不只是问答样例，而是可以被 `evaluation/run_eval.py` 自动评测的 benchmark。

### 3. 扩充测试集发现的问题

先跑 a001、a002 两条 smoke eval 时，发现 a002 夜游问题误命中了原来的 `guilin_city_day_plan` 模板。

原因是原模板匹配条件过宽：

```text
桂林 + 路线/规划/安排 + 交通/票价
```

夜游问题也包含这些词，所以被误判成桂林市区一日游。

这个问题说明：任务模板配置化之后，模板匹配规则本身也需要通过 benchmark 来检验，不能只看单个 q012 样本。

### 4. 修复方式

我做了两类调整。

第一，收窄原 `guilin_city_day_plan` 模板：

```text
必须命中“一天”或“一日游”
排除“晚上”“夜游”“夜间”“桂林千古情”“千古情”
```

这样夜游问题不会再误命中市区一日游模板。

第二，新增两个配置模板：

```text
yangshuo_day_plan
guilin_night_tour_plan
```

新增后，模板匹配结果变成：

```text
q012 -> guilin_city_day_plan
a001 -> yangshuo_day_plan
a002 -> guilin_night_tour_plan
```

这说明配置化模板确实具备扩展能力，不再只是 q012 的单点规则。

先跑了 a001、a002 smoke eval：
